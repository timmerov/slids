compiler_v2 — TODO (side quests)

Outstanding bugs, reach goals, anomalies, and deferred items — things we
started but didn't finish, found but deferred, or want later but aren't on the
main plan. Everything here is OPEN; closed/landed items are removed (the main
quest's progress lives in plan.txt).


BUGS (found, deferred)

  * BAKEDIMS CLOBBERS ALIAS RESOLUTION — constfold's bakeNodeDims rebuilds an
    array's type spelling from `n.return_type` (widen::spellOrEmpty), which is the
    GRAMMAR spelling with the element alias UNRESOLVED (e.g. `Integer[1]`), then
    stamps the rebuilt `Integer[3]` onto the entry — clobbering the `int[3]`
    resolve had already substituted. So an alias-element array with a
    const-EXPRESSION dim (`alias Integer = int; Integer nums[N];`) ends up typed
    `Integer[...]` (alias un-stripped), and a downstream use (`Integer[] it =
    ^nums[0]`) hits a spurious "Cannot implicitly cast 'Integer[]' to 'Integer[]'".
    A LITERAL-dim alias array is fine (the dim bakes at grammar, resolve strips the
    whole spelling, bakeNodeDims early-returns on empty dim_exprs). Exposed (then
    side-stepped) while fixing the literal-zero-dim check: routing literal dims
    through dim_exprs made EVERY alias array hit it, so the fix stayed at the
    grammar instead. Real fix: bakeNodeDims should rebuild onto the ALIAS-RESOLVED
    base — read the entry's slids_type (resolve's output) for the base type, not
    n.return_type — or have resolve re-resolve n.return_type post-bake.
    prerequisite: none; small, but touches the constfold dim-baking path.

  * ORPHAN-CHECK vs IMPORT FORWARD-DECLS — resolve's Pass 3 errors on any
    function declared-but-never-defined ("Function 'X' is declared but never
    defined.") — but a `.slh` header legitimately declares-only (defined in the
    adjacent .sl, or a link error). Today every declared-nowhere-defined
    function is an error regardless of file. Needs the `.slh` declare-only
    carve-out; lands with the cross-TU / .slh-propagation work (plan Phase 8).
    prerequisite: group with general work on import.

  * ARRAY PARAMETER NEEDS AN ALIAS — a function parameter can't carry array
    dimensions on the NAME: `int f(int a[3])` -> "Expected ',' or ')' in parameter
    list". The parameter parser stops after `Type name` and never runs the post-name
    `[N]` dim loop that the var-decl path has (grammar.cpp:1085+). A direct array-
    TYPE param (`int[3] a`) is rejected by the name-anchored declaration rule
    (reject_array_dims), so today the ONLY way to take an array param is via an
    alias (`alias A3 = int[3]; int f(A3 a)`). Found while adding array-param/return
    test coverage (the alias form + array RETURN `int[3] f()` both work). Fix: thread
    the name-dims parsing into the parameter parser, mirroring the var-decl path, so
    `int f(int a[3])` parses to an int[3] param. prerequisite: none.

  * COMPOSED-TYPE INDEXING DOESN'T RECURSE (alias-element arrays; array->tuple->
    array chains) — `emitElementAddr` (codegen) dispatches ONCE on the base type
    (array OR tuple) and applies the WHOLE index chain as that one kind, instead of
    walking the chain segment-by-segment and re-dispatching on the CURRENT type. Two
    symptoms, ONE root cause:
      - alias-element arrays: `alias Vec2 = int[2]; Vec2 va[3]` is a NESTED
        kArray(elem=int[2], dims=[3]) — rank 1 as the indexer counts it — so
        `va[i][j]` over-indexes ("indexes past the last dimension"). (A name-anchored
        `int va[3][2]` is a FLAT kArray(int,[3,2]) and works.)
      - array->tuple->array: `(int[3],int[4]) t[2]` then `t[i][slot][k]` — t[i] is an
        array index (element a tuple), [slot] a tuple slot (an array), [k] an array
        index. The base t is an array, so all 3 indices are applied as t's array dims
        (dims=[2], rank 1) -> over-index.
    USER FEEDBACK: once again, claude is lobbying for a hack instead of addressing
    the root cause. v1 had this bug; sad it reappears in v2. The proposed flatten-
    array-of-array at internArray (normalize `Vec2 va[3]` -> int[3][2]) is a HACK:
    it destroys the alias name and only fixes the alias-element symptom, leaving the
    array->tuple->array chain broken.
    TWO DISTINCT BUGS, both in codegen's emitElementAddr:
      (A) you don't know how many dimensions a type has — the RANK count is
          `get(strip(T)).dims.size()` (codegen.cpp:700), the TOP-LEVEL dims vector
          only, so a nested/composed type undercounts (Vec2 va[3]: reads 1, real
          rank 2) and the `chain.size() > dims` check fires a FALSE over-index. NOTE:
          classify does NOT have this bug — arrayElementType (classify.cpp:180) walks
          ONE level per subscript, recursing into the element, so it types va[i][j]
          and t[i][slot][k] correctly. The codegen rank check is a misplaced, buggy
          DUPLICATE of a semantic check classify already does right.
      (B) you don't know how to index those dimensions — emitElementAddr dispatches
          ONCE on the base type and applies the whole chain as that one kind, so it
          can't emit a mixed GEP chain (array dim -> struct slot -> array dim). (A)
          masks (B): fix the count alone and the GEP would then MIS-address.
    ROOT CAUSE FIX: make emitElementAddr walk the index chain innermost->outermost
    the way classify already walks the TYPE — track the current type, re-dispatch
    per segment (array dim -> array GEP; tuple slot -> struct GEP), advance to the
    resulting element type, continue. Both the bound/rank check and the GEP emission
    fall out of that walk; handles arbitrary composition; resolves BOTH symptoms.
    prerequisite: none; touches the index-addressing walk (codegen.cpp:636+).

  * ARRAY = TUPLE-VALUE EMITS INVALID LL — `int a[3]; a = tupleVar;` (a whole-array
    assign from a tuple VALUE, not a literal) is ACCEPTED by slidsc but emits
    invalid IR: the tuple is an LLVM struct `{i32,i32,i32}` and the array is
    `[3 x i32]`, and codegen whole-value load+stores across the mismatch —
    `store { i32,i32,i32 } %v, ptr %a` into a `[3 x i32]` alloca, which llc
    rejects ("defined with type '{ i32, i32, i32 }' but expected '[3 x i32]'").
    The "ARRAY FROM A TUPLE VALUE" reach goal below WRONGLY claims this "falls to
    the normal path + rejects" — it does NOT reject, it mis-emits (violates the
    no-invalid-LL rule). Same root for `tuple = arrayVar` (the reverse). FIX (now,
    independent of the reach goal): classify must REJECT an array<->tuple whole
    assignment whose two sides have different forms until the value-form copy is
    actually implemented. The array-from-tuple path only gates on a kTupleExpr
    LITERAL (isArrayFromTuple), so a tuple VALUE slips past into the generic
    same-"type"? assignment with no shape/representation bridge. prerequisite: none.


REACH GOALS

  * WHOLE-ARRAY MOVE / SWAP — `dst <-- arr` / `a <--> b` where the operands are
    ENTIRE arrays (not elements). moveswap.sl marks it "to be revisited". Move /
    swap LANDED for primitives / pointers / tuples (element operands `arr[i]` work);
    a whole array needs a per-element walk (swap = element-wise SSA exchange or a
    memcpy-through-temp; move = copy + null each pointer element). Also the null
    walk (codegen emitNullLeaves) does NOT yet recurse into an ARRAY leaf — wire
    that here too (today an array can't be a tuple slot, so it can't arise).
    prerequisite: the homogeneous-array machinery (LANDED). See [[project_v2_move_swap]].

  * ARRAY-FROM-TUPLE, SUB-ARRAY ASSIGN — `td[1] = (100,101)` assigns one ROW of
    a multi-dim array from a tuple. Needs a PARTIAL-index lvalue (a sub-array
    slice), which today errors "An array subscript must index every dimension."
    Same bucket as nested-for over a 2-D array (a partial index yields a
    sub-array). Deferred from the array-from-tuple landing (plan Phase 4, which
    did 1-D init/assign + multi-dim init/assign). prerequisite: partial-index
    array lvalues.

  * FOR-TUPLE OVER CLASS ELEMENTS — `for (p : classTuple)` where the element is a
    CLASS. The forced-by-ref rule for non-primitive elements has LANDED: a
    tuple/array/slid element forces the loop var to `T^` (classify), and a declared
    by-value loop var over one errors "must use a reference loop variable" — so
    NESTED tuples (`for (sub : tupleOfTuples) { for (x : sub^) ... }`) work today.
    What remains needs classes: by-VALUE iteration over a class element (a copy
    needs a constructor — not available) + the class copy/move machinery, and
    by-ref iteration once class-typed tuples exist. prerequisite: classes (Phase 5).

  * ARRAY FROM A TUPLE VALUE — `a = tupleVar` / `int a[3] = tupleVar` where the
    rhs is a tuple VARIABLE, not a literal. The landed array-from-tuple gates on a
    tuple LITERAL (kTupleExpr) so it can flatten the leaf NODES + elide the
    aggregate; a tuple value falls to the normal path and is NOT rejected — it
    mis-emits invalid LL (see the BUG above, "ARRAY = TUPLE-VALUE EMITS INVALID
    LL"; the rejection is the immediate fix, this is the feature). To support it:
    detect (array lvalue, tuple-typed value rhs), then flatten via extractvalue
    per slot (recursing nested tuples) and store each into the array slots — no
    node-level leaves to elide, so it copies through the value.
    prerequisite: the homogeneous array-from-tuple (LANDED).

  * ARRAYS OF TUPLES — `(int,int) arr[3] = ((1,2),(3,4),(5,6))`: an array whose
    ELEMENT type is a tuple. A DIFFERENT mapping than the homogeneous flatten —
    the outer tuple maps to elements, each kept as a tuple aggregate (no flatten).
    Indexing is `arr[array-index][slot-index]` (standard order — the array
    subscript outer, the tuple slot inner; array.sl spec block `six = arr[2][1]`
    / the comma `arr[1,2]`). Deferred from the array-from-tuple landing.
    prerequisite: the homogeneous array-from-tuple (LANDED) + element-is-a-tuple
    layout.

  * DROP THE ENUM QUALIFIER IN CONTEXT — when a variable's declared type is an
    enum, a bare member name on the rhs should resolve against that enum's
    namespace: `Language best = kRust;` then `best = kSlids;` (an implied
    `alias Language;` scoped to the statement / variable). Today bare `kRust` is
    "needs a namespace qualifier"; the canon block in enum.sl is commented
    pending this. Needs type-directed name resolution (the lvalue/assignment
    target type steering the rhs lookup) — no other feature needs it yet.
    prerequisite: lvalues.


DEFERRED FEATURES (later; may not be on the plan)

  * CLASS / USER-TYPE CONVERSION TARGET (Phase 5) — `(Type = expr)` value
    conversions LANDED for primitives (kConvertExpr; see [[project_v2_type_conversions]]).
    Two slices wait on classes:
      - a CLASS target `(String = "...")` / `(Class = Type^)` is class
        assignment-conversion (routes through the class's `op=` / char[]-assign
        path, exactly like `String s = "x"`), plus a pointer SOURCE to a class
        that defines `op=(Type^)`.
      - a USER-NAMED type as the target: the grammar only opens a conversion on a
        primitive type KEYWORD after `(`, so `(Foo = v)` isn't recognized yet
        (falls to parseExpr -> "Expected ')'"). Needs the is-this-a-type
        lookahead Phase 5 brings.
    typeconv.sl keeps the `(String=...)` example commented. prerequisite: classes.

  * CONVERSION OF AN OUT-OF-INT64-RANGE FLOAT LITERAL — tryFoldConvert does NOT
    fold a float-literal operand whose truncation falls outside int64 (|v| >=
    2^63); `(int = 1e30)` is lowered at runtime where `fptosi` of an out-of-range
    float is poison (C UB). Low value; emit a compile-time diagnostic if it bites.

  * CONST-POINTER CASTS (Phase 6) — the const-correctness slice of cast.sl is
    deferred (the A1 landing): `const Type^` (a const-qualified pointer type),
    the `<const>` / `<mutable>` qualifier-only casts, and the mutable<->const
    cast direction (implicit mutable->const, explicit const->mutable). v2 models
    const as an entry KIND, not a tracked TYPE qualifier on pointers, so none of
    these can be enforced yet. The rules are written + commented in cast.sl's
    canon block (lines ~21, 43-54, 77-86); land them with Phase 6 when const is
    a real qualifier. Until then those lines stay commented and untested.

  * FOR-CLASS — the last colon-form for shape (ranged/array/tuple/enum all LANDED
    and now lower in the DESUGAR stage). for-class needs classes + a for-each
    protocol (op[]/size or begin/end/next, Phase 5). Lowers to a kForLongStmt in
    desugar like the others. Not landable until classes exist.

  * FOR-ARRAY BY-VALUE NON-PRIMITIVE ELEMENT — the canon says by-value iteration
    requires a primitive element type (a reference works for all). With only
    primitive element types today the rule never fires, so it isn't enforced;
    add the check when slid/class array elements exist (Phase 5).

  * NAMED-CONST-TRUE LOOP NOT FLAGGED — the constant-true-loop special case
    (3B revisited: a constant-true loop with no break is non-completing — its
    after-code is unreachable + it is a return-terminator) is detected
    SYNTACTICALLY in resolve (a bool/int/uint/char literal condition, incl. the
    synthesized empty `()`), because resolve runs before constfold. A loop whose
    condition is a NAMED const that folds to true (`const t = true; while (t)`) is
    therefore not flagged — it keeps the possibly-completing behavior (no
    unreachable-after, a trailing return is still demanded).
    NOT a small cleanup (investigated 2026): a safe fix needs BOTH halves, both
    in classify (post-constfold, where constTruth sees the folded named const):
      - set Node.non_completing for a const-true-no-break loop (so endsInReturnNode
        stops the trailing-return over-demand AND codegen emits the `unreachable`
        exit — codegen already reads non_completing, propagated via desugar).
      - a NEW classify-side unreachable-after pass: flag a statement following a
        non-completing loop "Unreachable statement." Without it, codegen's block
        loop emits ALL children, so code after the loop lands after the
        `unreachable` -> INVALID IR. (resolve's 2A does this syntactically; classify
        must mirror it value-aware. Only fires for the named-const case — the
        syntactic case already errored in resolve and never reaches classify.)
    Plus a minor wart: resolve's possibly-zero DA may report "use of uninitialized"
    on the (now-unreachable) after-code before classify flags it. Low value
    (`while (true)` / empty-cond are the common forms and fully handled), real cost.

  * ##type — LANDED: the 7-macro set + `##type(x)`; the alias-label channel (an
    alias/enum-typed VARIABLE reports the source spelling — `Integer x` ->
    `Integer`, `Dir d` -> `Dir` — via a parallel alias_label, not un-erase); and
    the const qualifier (`const int8 a` -> `const int8`; a const read inside any
    expression STRIPS const). A bare/qualified TYPE NAME operand (alias / enum /
    qualified enum type) resolves to its underlying, and an enum MEMBER reports
    `const Enum` — both LANDED. See [[reference_stringify]]. OPEN: an aliased-
    FUNCTION `##type(Fn)` (waits on function aliases), and the `#` 5-tuple
    (Phase 5, blocked on tuples + `^`).

  * STRONG-CONST WIDENING ENFORCEMENT — LANDED at every assignment site. A strong
    const is a TYPED value (widen within family; REJECT narrowing / sign-change),
    not a flexing literal: `const int sc=5; int8 a=sc;` errors "Cannot implicitly
    narrow 'int' to 'int8'" like an int variable would. codegen's literal path only
    range-checks, so the check is in CLASSIFY — checkStrongConstAssign mirrors
    widen::convert's accept/reject + wording for the same-/cross-sign integer and
    float cases (cross-family / bool-source left to codegen, no double-report).
    Wired at var-decl-init, assign, store (`ref^=`/`arr[i]=`), return, and call-arg
    (BOTH the single-candidate and overload arg loops in classifyCall). Weak
    literals still flex. Canon: constant.sl kNarrow + neg_return/neg_arg/neg_store
    + kWide (widen positive).
    NOT enforced (a SEPARATE, broader matter — strong consts flexing as BINARY
    OPERANDS, not an assignment): `flexBinaryOperands` still flexes a strong-const
    literal into the partner's type, so `int8 a; a += sc` (sc strong int) treats sc
    as int8 (and `int8 v; v + sc` types as int8), where the typed-value model would
    make it int and reject the narrowing back into int8. Changing this means
    "a strong const does not flex in a binary op" — it re-types every binary expr
    with a strong-const operand, so it's its own design call, not an assignment-site
    fix. Decide separately.

  * ALIAS FUNCTION-RENAME — `alias Fn = LongName;` (use case: importing C
    functions with ugly names).

  * TEMPLATED ALIAS — `alias X<...> = T;`.

  * NESTED FUNCTION RESTRICTIONS TO REVISIT — three deliberate limits, each a
    compile error today (canon nested.sl says "needs to be revisited"):
      - DEEP NESTING — a nested function may not contain a further nested
        function ("Nested functions may not contain further nested functions.").
      - SIBLING CALLS — one nested function may not call another ("Calling a
        sibling nested function is not supported."). Lambda-lift threads only the
        host's captures, not a sibling's capture environment.
      - NESTED OVERLOADING — nested functions do NOT overload; a same-name nested
        pair is a "Duplicate declaration". classifyCall restricts candidates to
        file scope (parent_frame_id == 0). Revisit if nested overloads are wanted.


DEFERRED TESTS

  * DUP-DEF / DUP-DECL NOTE CARET POSITION — the "first defined here" note carets
    the first DEFINITION (Entry.def_tok) and "first declared here" the first
    DECLARATION (Entry.tok); verified by manual probe only. The //-EXPECT-ERROR
    runner matches a message SUBSTRING, not caret line/col, so this attribution
    can't be locked by a .sl negative. Needs a caret-aware assertion the harness
    doesn't have.

  * IF-CONDITION TYPE DIAGNOSTIC — classify rejects an if condition whose type
    isn't coercible to bool ("An if condition must be a condition expression;
    type 'X' is not."), but no constructible expression has such a type yet:
    numerics + char[] coerce, and a void-typed condition needs a void function,
    which is unbuildable until bare `return;` lands. The check is defensive and
    UNTESTED; add a negative once a non-coercible value exists. prerequisite:
    void functions (bare return) or slid/class values.

  * `^`-REFERENCE CONDITION — isCoercibleToBool (shared with `!`/`&&`/`||`)
    matches only the `[]` iterator suffix, not `^` references, so `if (ref)`
    will wrongly error when `^` reference values land (Phase 4). Widen the
    shared helper then — at the helper, not the if-condition call site.
    prerequisite: pointers/iterators (Phase 4).
