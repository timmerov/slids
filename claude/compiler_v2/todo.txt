BUGS (found, deferred)

  * TUPLE TYPE WITH A CLASS SLOT — `(Cz, int) t` / `(Cz, int)` as any composite
    type position fails resolveDeclType with "Unknown type '(Cz, int)'.", even
    though resolveTypeRef walks tuple slots structurally and the class is
    registered. The structural walk produces the resolved slot handles fine but
    requireKnownType (resolve.cpp:98) rejects the composed type. Surfaced
    writing the deferred class-target conversion negative for typeconv.sl; same
    gap blocks tuple-with-class-slot field decls more generally. Needs the
    "known" predicate to walk the tuple slots and accept each-leaf-known instead
    of demanding the composed handle be marked known. Low-medium priority.

  * SELF-SWAP / SELF-MOVE FOR COMPLEX LVALUES — classify rejects a self-swap /
    self-move when both operands are the same BARE variable (isSameLvalue: two
    kIdentExpr with the same resolved_entry_id) -> "Cannot swap a value with itself"
    / "Cannot move a value onto itself". An indexed / deref self (`arr[i] <--> arr[i]`,
    `p^ <-- p^`) is NOT detected — that needs structural lvalue-equality (same base +
    same index/deref chain), and a runtime index can't be compared statically anyway.
    Low priority. Widen isSameLvalue when complex-lvalue equality is wanted.

  * constant tuples, arrays, classes - the error message is misleading.
    what exactly are there?
    unlike most constants, they're actually allocated and initialized.
    const int arr[3] = (1,2,3); an array of 3 const ints.
    const (int,int) tuple = (1,2); a not-mutable tuple of 2 ints.
    const String s = "hi"; a not-mutable string.


DEFERRED FEATURES (later; may not be on the plan)

  * FOR-ARRAY BY-VALUE NON-PRIMITIVE ELEMENT — by-value iteration requires a
    primitive element type (a reference works for all). Only the CLASS-element
    variant remains, when class array elements exist (Phase 5).

  * CLASS COPY-INIT AT A DECL — `Class a = sameClassValue` (initialize a class
    local from another value of the SAME class — e.g. the result of a function
    that returns a class by value) is mishandled: classify routes the decl through
    classifyClassInit, which treats the rhs as a FIELD-init argument, so a 1-field
    class reports "Cannot initialize field 'x_' ... from a value of type 'Class'."
    A plain ASSIGN `a = b` works (checkSlidAssign allows same-class), and
    `= scalar` / `= tuple` construct — only whole-class copy-INITIALIZATION at a
    decl is wrong. Surfaces when a nested function RETURNS a local class by value.
    Needs the class copy path (whole-value copy vs field-init). prerequisite: class
    copy/move.

  * METHODS — follow-ups (the statement-call form + plain-`=` field writes LANDED).
    - EXPRESSION-FORM CALL: a value-returning method used as a value
      (`int x = obj.m()`) doesn't parse; only the statement form `obj.m(args);` is
      wired. Needs the call shape in the expression parser (parsePostfix) + a
      kMethodCallExpr (or reuse) classified/lowered like the statement form.
    - COMPOUND FIELD WRITES in a member body: a BARE `x_ += 1` / `x_++` isn't
      field-rewritten (only plain `x_ = ...` is — buildSelfField on the assignment
      LHS). Hook the field-rewrite onto the aug-assign / inc-dec LHS so a bare
      `x_ += 1` rewrites to the self-field form (the complex-lvalue chain then
      lowers it).
    - SIBLING METHOD CALL VIA `self` (deferred to a later session): calling one
      method from another. Today `self.m()` / `self.m()` parse-errors ("Expected
      statement"), and a BARE `m()` now errors cleanly ("Method 'm' must be called
      on an object.") instead of crashing. Needs `self` as a usable receiver.

  * CLASS / USER-TYPE CONVERSION TARGET (Phase 5) — `(Type = expr)` LANDED for
    primitives, tuples, and arrays (lockstep recursive walk; see
    [[project_v2_type_conversions]]). Two slices still wait:
      - a CLASS target `(String = "...")` / `(Class = Type^)` is class
        assignment-conversion (routes through the class's `op=` / char[]-assign
        path, exactly like `String s = "x"`), plus a pointer SOURCE to a class
        that defines `op=(Type^)`. The classify defer arm exists ("conversion
        to a class is not yet implemented") but is unreachable from any test
        today — the user-named top-level form `(Foo=v)` doesn't parse, and a
        tuple/array with a class slot fails earlier at resolveDeclType with
        "Unknown type '(Cz, int)'" (a separate gap).
      - a USER-NAMED type as the TOP-LEVEL target: the grammar opens a
        conversion on a primitive type KEYWORD after `(` or on a balanced
        `((...)...)` shape (the tuple/array path), so `(Foo = v)` (single
        parens, identifier-led) isn't recognized; falls to parseExpr →
        "Expected ')'". `((Foo)=v)` (extra parens via the balanced-paren
        lookahead) DOES work via single-slot tuple-collapse + alias
        transparency, but only for primitive-backed aliases — a class-backed
        alias would hit the defer arm above. Needs the is-this-a-type lookahead
        Phase 5 brings.
    typeconv.sl keeps the `(String=...)` example commented. prerequisite: classes.

  * CLASS-FIELD INIT FROM A SIZE-1 TUPLE — the "vexing parse" (Phase 5). CONTEXT,
    the class structure: a class IS a named tuple. `Name(field-list){body}` interns
    a kSlid that CARRIES its field slot types (the layout rides on the type, so
    codegen needs no symbol table), a ClassInfo holds the field names/defaults
    (parse-side only), `.field` lowers to a slot index, and construction reuses the
    tuple aggregate path. Two init forms feed it: the CALL form
    `Name(arg0, arg1, ...)` — each arg is ONE field's value, so a tuple arg stays
    WHOLE and a class-typed field constructs from it recursively (its own defaults
    fill the rest) — and the `=` form `Name x = tuple` — the init tuple is the
    FIELD tuple, SPREAD slot-to-field. They coincide for a multi-field class
    (`= (6,7)` == `(6, 7)`) and DIVERGE when one argument is itself a tuple bound
    for a single (class) field: the call form keeps it whole, `=` spreads.
    THE GAP: a SIZE-1 init tuple is INEXPRESSIBLE in the `=` form — `( X )` is a
    grouping paren that COLLAPSES to `X` (size-1 tuples collapse to their slot
    everywhere). So `Outer x = ( (13,17) )` (Outer has one class field) collapses
    to `= (13,17)`, a 2-slot tuple spread into 1 field -> "1 field(s) but 2
    initializer(s)". No `=`-spelling passes a whole tuple to a lone field; the
    CALL form `Outer x( (13,17) )` is the only way (parseCallArgs preserves the
    argument boundary the collapsing parens destroy). PUNTED: use the call /
    initializer-list form for a single-field-from-tuple init; canon construct.sl
    keeps the `= ( (13,17) )` case commented with a "vexing parse" note. A real fix
    needs a non-collapsing 1-tuple syntax, which touches grouping parens globally —
    not worth it. prerequisite: a 1-tuple spelling (none planned).

  * CONST ENFORCEMENT (Phase 6) — const representation/syntax is in (qualifier,
    param munge, `mutable`, the casts); ENFORCEMENT is not. const is unchecked
    today, so a const->mutable drop is silently accepted. To do:
      - lvalue carries const / rvalue read is mutable (the value-category model);
      - const->mutable rejected at assign + call sites without an explicit `<mutable>`;
      - no write through a const reference (every assign-family site);
      - const-overload matcher (match-key strips const);
      - method-level const + a const receiver `_$recv`;
      - `mutable` REQUIRED on a move/swap operator param (`op<--`/`op<-->`) — also
        blocked on operator overloading existing (Phase 5+; op<-- does not parse).
    cast.sl's const-correctness lines + the deep-const local forms in mutable.sl
    stay syntax-only until this lands.

  * CONST `(const T)[N]` STATEMENT-LOCAL DECLARATOR GAP — a parenthesized const
    type with a SIZED-array suffix as a statement-level local (`(const int)[3] a;`,
    `(const int, int)[2] a;`) does not parse: the parenthesized-type-decl lookahead
    admits only `^` / `[]`, not `[N]`. Fine as a tuple SLOT (`((const int)[2], int)`)
    or a parameter. Pre-existing declarator gap, not const-specific. (Related: a
    leading `const` at a statement START routes to the named-CONSTANT path — for a
    SCALAR that's a substituted constant, but for a tuple/array/class it gives a
    MISLEADING error; those are allocated + initialized const-qualified VARIABLES —
    see the const-aggregate BUG above. A const-qualified scalar local is written in
    a type position, `(const int)^ r`.)

  * FOR-CLASS — needs classes + a for-each
    protocol (op[]/size or begin/end/next, Phase 5). Lowers to a kForLongStmt in
    desugar like the others. Not landable until classes exist.

  * NAMED-CONST-TRUE LOOP NOT FLAGGED — the constant-true-loop special case
    (3B revisited: a constant-true loop with no break is non-completing — its
    after-code is unreachable + it is a return-terminator) is detected
    SYNTACTICALLY in resolve (a bool/int/uint/char literal condition, incl. the
    synthesized empty `()`), because resolve runs before constfold. A loop whose
    condition is a NAMED const that folds to true (`const t = true; while (t)`) is
    therefore not flagged — it keeps the possibly-completing behavior (no
    unreachable-after, a trailing return is still demanded).
    NOT a small cleanup (investigated 2026): a safe fix needs BOTH halves, both
    in classify (post-constfold, where constTruth sees the folded named const):
      - set Node.non_completing for a const-true-no-break loop (so endsInReturnNode
        stops the trailing-return over-demand AND codegen emits the `unreachable`
        exit — codegen already reads non_completing, propagated via desugar).
      - a NEW classify-side unreachable-after pass: flag a statement following a
        non-completing loop "Unreachable statement." Without it, codegen's block
        loop emits ALL children, so code after the loop lands after the
        `unreachable` -> INVALID IR. (resolve's 2A does this syntactically; classify
        must mirror it value-aware. Only fires for the named-const case — the
        syntactic case already errored in resolve and never reaches classify.)
    Plus a minor wart: resolve's possibly-zero DA may report "use of uninitialized"
    on the (now-unreachable) after-code before classify flags it. Low value
    (`while (true)` / empty-cond are the common forms and fully handled), real cost.

  * ALIAS FUNCTION-RENAME — `alias Fn = LongName;` (use case: importing C
    functions with ugly names).

  * TEMPLATED ALIAS — `alias X<...> = T;`.

  * NESTED FUNCTION RESTRICTIONS TO REVISIT — three deliberate limits, each a
    compile error today (canon nested.sl says "needs to be revisited"):
      - DEEP NESTING — a nested function may not contain a further nested
        function ("Nested functions may not contain further nested functions.").
      - SIBLING CALLS — one nested function may not call another ("Calling a
        sibling nested function is not supported."). Lambda-lift threads only the
        host's captures, not a sibling's capture environment.
      - NESTED OVERLOADING — nested functions do NOT overload; a same-name nested
        pair is a "Duplicate declaration". classifyCall restricts candidates to
        file scope (parent_frame_id == 0). Revisit if nested overloads are wanted.

  * ORPHAN-CHECK vs IMPORT FORWARD-DECLS — resolve's Pass 3 errors on any
    function declared-but-never-defined ("Function 'X' is declared but never
    defined.") — but a `.slh` header legitimately declares-only (defined in the
    adjacent .sl, or a link error). Today every declared-nowhere-defined
    function is an error regardless of file. Needs the `.slh` declare-only
    carve-out; lands with the cross-TU / .slh-propagation work (plan Phase 8).
    prerequisite: group with general work on import.

  * UNUSED PARAMETER AS A COMPILE ERROR — NEEDS REVIEW. Today the unused-local
    sweep (resolve) covers only body-declared locals; parameters are exempt
    (vardecl.sl's `unused_param` compiles). Proposal: flag a parameter that is
    never read, like an unused local. NEEDS REVIEW — params are often legitimately
    unused (interface conformance, overload / signature symmetry, a callback that
    ignores an arg), so this likely wants an opt-out (a name convention or an
    explicit discard) rather than a blanket error. Decide the policy first.

  * x++ AS AN LVALUE (but NOT ++x) — the PPID model lifts the bump off the lvalue,
    so a POST-inc target is well-defined: `x++ = 42` -> `x = 42; x++` (store, THEN
    bump — the stored value is incremented to 43). A PRE-inc target is rejected:
    `++x = 42` would bump THEN store (`x = x+1; x = 42`), so the pre-bump is dead.
    So `x++` may be an lvalue; `++x` may not. This also makes chained `x++ ++` well
    defined (two bumps of x — see CHAIN POSTFIX PPID). Today parseIncDecStmt
    (grammar.cpp:1467) commits to the bare `x++;` statement the moment it sees
    `ident ++` and demands `;`, so the lvalue form never parses. Needs the
    statement/lvalue parse to accept a postfix-inc lvalue + desugar to lift the bump
    after the store.


DEFERRED TESTS

  * CTOR/DTOR THROUGH AN ARRAY / TUPLE class FIELD — an array/tuple-of-hook-class
    LOCAL works now (`CtorDtor arr[3] = (36,37,38)` runs ctors 36,37,38 / dtors
    38,37,36 — classify constructs each class element via constructClass; the
    needs-fixpoint + emitConstructHooks/emitDestructHooks descend array elem /
    tuple slots; the var-decl arm runs the hooks after the array field-init).

    classZeroValue / classifyClassInit only CONSTRUCT a direct kSlid field; an
    array/tuple field has no default-construction path (the needs-fixpoint already
    descends it, so the hooks WOULD fire once construction lands). Fix:
    classZeroValue builds an N-element array (each element default-constructed) /
    a per-slot tuple, routed through the array-from-tuple slot path. Tests go in
    array.sl / tuple.sl.

  * DESTRUCTION ON NON-BLOCK EXITS — every ctor/dtor test exits a scope by BLOCK
    fall-through. The destructor-balance invariant's return / break / continue arms
    (emitUnwindDtors) are implemented but UNTESTED: a hook instance at function
    scope destroyed at `return`, one inside a loop destroyed on `break`/`continue`,
    an early `return` out of a nested `if`. ALSO the loop-VARIABLE case (a class
    declared in a `for` varlist) — a known limitation: the varlist alloca hoists to
    entry, so the loop var registers in the ENCLOSING scope, not a per-iteration
    one, and isn't destroyed at loop exit (a loop-BODY local is fine). Tests go in
    flow/.

  * CTOR/DTOR CALLED EXACTLY ONCE UNDER COPY / MOVE — canon says the ctor/dtor run
    exactly once each; a copy / move / reassignment / by-value param-or-return of a
    hook class must not double- or skip-run them. Needs class copy/move (not built).
    Test in moveswap.sl.

  * DUP-DEF / DUP-DECL NOTE CARET POSITION — the "first defined here" note carets
    the first DEFINITION (Entry.def_tok) and "first declared here" the first
    DECLARATION (Entry.tok); verified by manual probe only. The //-EXPECT-ERROR
    runner matches a message SUBSTRING, not caret line/col, so this attribution
    can't be locked by a .sl negative. Needs a caret-aware assertion the harness
    doesn't have.

  * IF-CONDITION TYPE DIAGNOSTIC — classify rejects an if condition whose type
    isn't coercible to bool ("An if condition must be a condition expression;
    type 'X' is not."), but no constructible expression has such a type yet:
    numerics + char[] coerce, and a void-typed condition needs a void function,
    which is unbuildable until bare `return;` lands. The check is defensive and
    UNTESTED; add a negative once a non-coercible value exists. prerequisite:
    void functions (bare return) or slid/class values.

  * `^`-REFERENCE CONDITION — isCoercibleToBool (shared with `!`/`&&`/`||`)
    matches only the `[]` iterator suffix, not `^` references, so `if (ref)`
    will wrongly error when `^` reference values land (Phase 4). Widen the
    shared helper then — at the helper, not the if-condition call site.
    prerequisite: pointers/iterators (Phase 4).

  * FOR-TUPLE OVER CLASS ELEMENTS — What remains needs classes: by-VALUE iteration over a class element (a copy
    needs a constructor — not available) + the class copy/move machinery, and
    by-ref iteration once class-typed tuples exist. prerequisite: classes (Phase 5).


REACH GOALS

  * DROP THE ENUM QUALIFIER IN CONTEXT — when a variable's declared type is an
    enum, a bare member name on the rhs should resolve against that enum's
    namespace: `Language best = kRust;` then `best = kSlids;` (an implied
    `alias Language;` scoped to the statement / variable). Today bare `kRust` is
    "needs a namespace qualifier"; the canon block in enum.sl is commented
    pending this. Needs type-directed name resolution (the lvalue/assignment
    target type steering the rhs lookup) — no other feature needs it yet.
    prerequisite: lvalues.

  * ACCESS A SHADOWED LOCAL CLASS — a class defined in a function body may be
    shadowed by a same-named class in a nested function (allowed; bare lookup is
    innermost-first, so the inner one wins and the outer becomes unreachable — see
    nested_cls.sl). There is no syntax to name the shadowed (enclosing-scope) one.
    `::` is NOT it: `::` starts resolution at GLOBAL scope, but a local class lives
    in its function's body frame, not file scope, so `::` overshoots it. Nor can
    `::main:Function` work — `main` is a function, not a namespace, so it can't be
    a qualifier segment. The meaning of `::` is fixed (global) and will not change.
    Reaching an outer shadowed local class would need a DISTINCT "enclosing scope"
    qualifier, with its multi-level semantics (skip-one vs nearest-match-outward)
    still to be decided. No current need; the shadowed class is simply accepted as
    unreachable. prerequisite: local classes in function bodies.


ASPIRATIONAL FEATURES (someday; low priority, may never land)

  * REOPEN ENUMS — re-`enum SameName ( ... )` to add members to an existing
    enum (member names can't collide across openings). Open spec qs: does the
    value counter restart at 0 on reopen, and must the first member of a
    reopening carry an explicit value? Defer until decided. (Namespaces already
    reopen; enums would reuse openNamespace's find-or-create, but the
    value-counter continuation across openings is the undecided part.)

  * NAMED-SWITCH BREAK (`break someswitch;`) — breaking a non-innermost switch by
    its name. switch.sl marks it "distant future, low utility". switch naming was
    deliberately left out. Needs switch blocks to carry a label (Node.label +
    parse the `:name` after a switch body, currently a parse error) and the
    numbered/named resolver to consider switch frames as named targets.

  * CHAIN POSTFIX PPID — repeated postfix inc/dec, `x++ ++ ++` (and the lexer's
    greedy `x+++++` -> `x ++ ++ +`). WELL-DEFINED in the PPID model (the bump lifts
    off the lvalue, so `x++ ++` is two bumps of x — NOT the C `(x++)++`-needs-an-
    lvalue problem; slids doesn't evaluate-then-rebump). Blocked GRAMMATICALLY, not
    semantically: parsePostfix applies ONE postfix ++/-- and returns (grammar.cpp:
    818), so a second `++` is a parse error. Low priority / low utility. Prereq:
    `x++ AS AN LVALUE` (above) — same lift-the-bump-off-the-lvalue machinery. Needs
    parsePostfix to LOOP ++/--.
