compiler_v2 — TODO (side quests)

Outstanding bugs, reach goals, anomalies, and deferred items — things we
started but didn't finish, found but deferred, or want later but aren't on the
main plan. Everything here is OPEN; closed/landed items are removed (the main

BUGS (found, deferred)

  * CHAR-ARITH LOSES CHAR-NESS — int-class binary arith folds every operand to
    signed int64 and emits kIntLiteral, dropping a char operand's kind:
    `'A' + 1` folds to int 66, not char 'B'. So an auto-incremented char enum
    member prints numerically: `enum char Key ( a = 'A', b )` gives `b = 66`,
    while an explicit `b = 'B'` prints `B`. foldIntArithBitwise
    (constfold.cpp) returns kIntLiteral for every non-overflowing + - *; the
    UNARY fold already preserves operand kind per rule 1e, binary does not.
    Fix needs the binary-result type rule for mixed/char operands decided
    (char+int -> char? slids widen rules would dictate). No invalid IR; just a
    surprising print. prerequisite: ##type (the char-vs-int result distinction
    is unobservable until a value can report its own type).

  * QUALIFIED-TYPE CARET GRANULARITY — a bad segment in a namespace-qualified
    TYPE spelling (`A:B:C x`) carets at the whole type position, not the
    offending segment. resolveQualifiedType (resolve.cpp) joins the spelling
    into a flat string with no per-segment tokens (unlike a kIdentExpr, which
    carries qualifier_toks), so `toks(segs.size(), tok)` points every segment at
    the same decl-type token. Attribution is correct file/line (never file_id=0),
    just coarse. A B-precise caret would need parseType to thread per-segment
    tokens through the type spelling (or a structured qualified-type node).
    No invalid IR; diagnostic precision only.

  * ORPHAN-CHECK vs IMPORT FORWARD-DECLS — resolve's Pass 3 errors on any
    function declared-but-never-defined ("Function 'X' is declared but never
    defined.") — but a `.slh` header legitimately declares-only (defined in the
    adjacent .sl, or a link error). Today every declared-nowhere-defined
    function is an error regardless of file. Needs the `.slh` declare-only
    carve-out; lands with the cross-TU / .slh-propagation work (plan Phase 8).
    prerequisite: group with general work on import.


REACH GOALS

  * DROP THE ENUM QUALIFIER IN CONTEXT — when a variable's declared type is an
    enum, a bare member name on the rhs should resolve against that enum's
    namespace: `Language best = kRust;` then `best = kSlids;` (an implied
    `alias Language;` scoped to the statement / variable). Today bare `kRust` is
    "needs a namespace qualifier"; the canon block in enum.sl is commented
    pending this. Needs type-directed name resolution (the lvalue/assignment
    target type steering the rhs lookup) — no other feature needs it yet.
    prerequisite: lvalues.


DEFERRED FEATURES (later; may not be on the plan)

  * FOR TYPELESS / REUSE LOOP VAR — a typeless loop var in any for form:
    `for (x = 0) ...` (long), `for (x : 0..10)` (ranged), `for (x : Enum)` (enum).
    Reuse an enclosing-scope `x` if one exists, else declare a fresh inferred-type
    loop var (type from the initializer / range element / enum). Needs inferred-
    init. Today all for forms require an explicit var type; a typeless entry hits
    a cryptic parse error (parseType eats the name as a type). The reuse rule
    (`int row = 0; for (row : 0..8)` writes the outer row, holding the last value
    after) fixes the v1.md chess-sample bug. On the Session-4 plan (plan-flow.txt)
    — land with inferred-init.

  * FOR-ARRAY / FOR-TUPLE / FOR-CLASS — the remaining colon-form for shapes,
    blocked on the underlying types: arrays + anon-tuples (Phase 4) and classes
    with the for-each protocol op[]/size or begin/end/next (Phase 5). Each
    desugars to a kForLongStmt over an index (like v1's _finishIter). Not landable
    until those types exist.

  * RANGED-FOR LITERAL-FOLD LOOP-FOREVER CHECK — a ranged-for `for (var : start ..
    cmp end op step)` whose start/end/step are ALL integer literals can be checked
    at compile time for a guaranteed non-terminating (or zero-iteration-degenerate)
    loop: step == 0; start == end under a strict cmp; cmp-direction vs step-sign
    mismatch (e.g. cmp `<` ascending but a `-`/negative step, like `0 .. 10 + -1`).
    Designed in v1 but NEVER built (v1.md "For-loop syntax — remaining work"); v1
    and v2 both just spin at runtime today. In v2 this fits constfold (it already
    folds all-literal subtrees) — flag on a kForLongStmt whose synthesized cond +
    step fold to constants. Literal-only by design; runtime operands (`0 .. n*2`,
    `0 .. != x`) are not caught. Lands with / after ranged-for.

  * CONSTANT-TRUE LOOP -> UNREACHABLE AFTER-CODE — a `while (true)` / `while {} ()`
    with no break never exits, so code after it is unreachable; likewise an empty
    post-condition. NOT flagged today: per 3B there is no constant-true loop
    special case (the loop is modeled possibly-zero / always-completing). The
    constant-condition unreachable check (classify) covers if-true/if-false and
    while-FALSE bodies, but deliberately skips constant-true loops. Revisit only
    if 3B is revisited.

  * REOPEN ENUMS — re-`enum SameName ( ... )` to add members to an existing
    enum (member names can't collide across openings). Open spec qs: does the
    value counter restart at 0 on reopen, and must the first member of a
    reopening carry an explicit value? Defer until decided. (Namespaces already
    reopen; enums would reuse openNamespace's find-or-create, but the
    value-counter continuation across openings is the undecided part.)

  * ##type — LANDED: the 7-macro set + `##type(x)`; the alias-label channel (an
    alias/enum-typed VARIABLE reports the source spelling — `Integer x` ->
    `Integer`, `Dir d` -> `Dir` — via a parallel alias_label, not un-erase); and
    the const qualifier (`const int8 a` -> `const int8`; a const read inside any
    expression STRIPS const). See [[reference_stringify]]. OPEN: the enum-MEMBER
    label (next item), and the `#` 5-tuple (Phase 5, blocked on tuples + `^`).

  * ENUM MEMBER LABEL — `##type(Dir:kNorth)` should be `const Dir`, not the
    current `const int`. A named-enum member's kConst entry is typed with the
    bare underlying and carries no alias_label (registerEnumMembers, resolve.cpp:
    `e.slids_type = node.return_type`; e.alias_label never set), so `##type` of a
    bare member gets the const qualifier right (the member IS a const) but reports
    the underlying `int`, not the enum name. A VARIABLE of the enum type already
    works (`Dir d` captures alias_label "Dir" at its decl, via resolveDeclType
    erasure). The bare-reference mechanism is one line: stamp
    `e.alias_label = node.name` for a named enum (anonymous enum -> no name -> no
    label), matching the desugaring `enum Dir (kNorth)` -> `Dir { const Dir
    kNorth = 0; }`.
    consequences: the one-line stamp also turns ON enum-label PROPAGATION through
    the existing binaryLabel rule — `Dir:kNorth + 1` would then carry "Dir"
    (alias+literal -> alias, C-style "enum + offset stays the enum"). That forces
    the named-type binary-RESULT rule to be decided: is `kNorth + kSouth` a Dir? a
    Dir-typed loop var stepping past the last member? It is the SAME binary-result
    rule the CHAR-ARITH-LOSES-CHAR-NESS bug needs. Also couples with DROP-THE-ENUM-
    QUALIFIER (a Dir-typed lvalue steering bare-member lookup) and with const-strip
    (would `##type(Dir:kNorth + 1)` keep "Dir"?).
    prerequisite: alias-label channel (LANDED) + a decided binary-result
    label-propagation rule for named types (shared with char-arith). That decision
    is the real gate — not the one-line stamp.
    schedule: land grouped with the CHAR-ARITH-LOSES-CHAR-NESS fix + the enum-label
    propagation decision (one binary-result rule serves all three); sequenced AFTER
    inferred-init (current focus), BEFORE Phase 5 classes. The bare-reference stamp
    alone is small and could land sooner once the propagation rule is settled.

  * ALIAS FUNCTION-RENAME — `alias Fn = LongName;` (use case: importing C
    functions with ugly names).

  * TEMPLATED ALIAS — `alias X<...> = T;`.


DEFERRED TESTS

  * IF-CONDITION TYPE DIAGNOSTIC — classify rejects an if condition whose type
    isn't coercible to bool ("An if condition must be a condition expression;
    type 'X' is not."), but no constructible expression has such a type yet:
    numerics + char[] coerce, and a void-typed condition needs a void function,
    which is unbuildable until bare `return;` lands. The check is defensive and
    UNTESTED; add a negative once a non-coercible value exists. prerequisite:
    void functions (bare return) or slid/class values.

  * `^`-REFERENCE CONDITION — isCoercibleToBool (shared with `!`/`&&`/`||`)
    matches only the `[]` iterator suffix, not `^` references, so `if (ref)`
    will wrongly error when `^` reference values land (Phase 4). Widen the
    shared helper then — at the helper, not the if-condition call site.
    prerequisite: pointers/iterators (Phase 4).
