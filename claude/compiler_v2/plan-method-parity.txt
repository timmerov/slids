================================================================================
1. THE IRREDUCIBLE DIFFERENCE (why this is "shared engine", not "one path")
================================================================================

  A free function is found by NAME + SCOPE (file/namespace + qualifier). A method
  is found by RECEIVER TYPE — strip the receiver to its class, look in that class's
  member frame. The lookup KEY differs and cannot be collapsed: a method call
  starts from an object, a function call from a bare name. So the target is a
  shared ENGINE with two thin front-ends, NOT a single code path. Method-only
  concerns a unified path could never erase: `self` binding + the bare-field ->
  `self.field` rewrite (resolve), the class-scoped symbol mint (desugar), the const
  receiver munge (Phase 6), and virtual vtable dispatch (Phase 7). Construction
  (`Class(args)` -> classifyClassInit) and overloadable operators are ADDITIONAL
  front-ends on the same engine, out of scope here.


================================================================================
2. THE SHARED ENGINE
================================================================================

  Extract from classifyCall (the ranking loop + viability + tie/no-match, lines
  ~1748-1788) a reusable core:

    struct OverloadResult { int best; bool ambiguous; bool none; };
    OverloadResult resolveOverload(tree, cands, args, recv_offset, diag);

  - `cands` — candidate entry ids (same-name, from the front-end's gather).
  - `args` — the provided argument nodes (already inferred without context).
  - `recv_offset` — 0 for a free function; 1 for a method (params[0] = `_$recv`,
    the receiver, always an exact match — rank over params[1+recv_offset..] vs
    args, and the arity RANGE is [num_required - recv_offset .. param_count -
    recv_offset]).

  fillDefaults (classify.cpp:1696) gains the same recv_offset so a method's
  defaults fill from index nargs+recv_offset. Per-arg type-check (inferExpr with
  the param as context + checkValueAssign) is shared too — it is identical in both
  paths modulo the offset.


================================================================================
3. STAGE-BY-STAGE CHANGES
================================================================================

  GRAMMAR   — none. `_(){}` / methods already inject `_$recv`; the definition is a
              kFunctionDef.
  RESOLVE   — allow a method OVERLOAD SET in a class member frame: relax the
              member dup-decl check so METHOD-vs-METHOD (kFunction-vs-kFunction,
              same name) is permitted, while still rejecting a method that collides
              with a FIELD, a const/alias/enum member, or a nested CLASS of the same
              name. (Mirror the file-scope multi-entry rule, scoped to the class
              frame.) The two lookup front-ends stay: name-scope (functions) and
              receiver-frame (methods) — both now produce a SET, not a single id.
              Sibling/`self.f()` calls already rewrite to method calls, so they ride
              the same path for free.
  CONSTFOLD — none (defaults already fold; methods are kFunction).
  CLASSIFY  — (1) factor classifyCall's ranking into resolveOverload (recv_offset
              0). (2) inferMethodCall (classify.cpp:2805): replace the single
              findMemberDeclared + hard arity check with a GATHER of all same-name
              kFunction members of the receiver's class frame, then resolveOverload
              (recv_offset 1) + fillDefaults. Ambiguity / no-match messages mirror
              the function wording ("Ambiguous call" / "No matching overload"),
              method-phrased.
  DESUGAR   — none beyond the existing class-scoped symbol mint (the chosen overload
              is just a different entry id; the symbol still comes from the method's
              DEFINING class).
  CODEGEN   — none. A method is already a lifted top-level function; the resolved
              overload id selects the signature exactly as a free-function overload
              does.


================================================================================
4. SCOPE
================================================================================

  IN:  named-method overloading, default params on methods, infer-from-default on
       method params; the shared resolveOverload + fillDefaults(recv_offset).
  NOT APPLICABLE — constructors have NO method-parity analog. A class's
  `Class(field-list)` header is a FIELD list; CONSTRUCTION = initialize the fields
  from a compatible tuple IN ORDER, then run the nullary `_()` hook (paired with the
  `~()` dtor at scope exit). `_()` takes NO author params and is not a signature-
  bearing callable, so there is nothing to overload and no "ctor params" exist — the
  `Class(args)` "args" are field INITIALIZERS (a tuple spread via classifyClassInit),
  not constructor arguments. The adjacent overloadable construct is POST-construction
  ASSIGNMENT (`op=`), tracked under operator overloading — not a constructor feature.
  (Field DEFAULTS — `Class(field = default)` — are an initialization concept, also
  separate from this work.)

  OUT (follow-ups, flagged):
    * const-overload (Phase 6 — the match-key strips const; a const + non-const
      method are distinct overloads): the engine must not preclude it.
    * inheritance: an overload set spanning base + derived (hiding vs overriding)
      and virtual dispatch (Phase 7). Leave as a known seam.
    * operator overloading (Phase 5, separate): another adapter on the same engine.
