compiler_v2 — PLAN (the main quest)

Feature implementation order. Items are open or LANDED; updated as we make
progress. Outstanding bugs / reach goals / deferred side-quests live in
todo.txt, not here. Detailed per-stage current-state notes (what each stage
does today, stage by stage) live in readme.txt.

Each phase adds primitives the next phase depends on. widen.cpp threads through
every conversion site as features land; treat each new expr/decl/assign site
as a widen call-out.

PIPELINE — current shape

    lex -> numeric -> grammar -> resolve -> constfold -> classify
        -> desugar -> optimize -> codegen

  resolve owns: parse::Entry vector population, frame push/pop, ident
  resolution (resolved_entry_id stamping), wrong-kind diagnostics
  (assign-to-function, call-a-variable), duplicate-decl + signature
  mismatch + arity-mismatch + multi-arg-print-intrinsic detection,
  type-spelling validation (widen::isKnownType for declared / return /
  parameter types), namespaces + enums (lowered to entries here), and the
  orphan-function check (declared-but-never-defined). Two passes per scope
  (collect decls then walk bodies) so forward references work.

  classify is type inference + (Phase 3) overload resolution. Stamps
  inferred_type / op_type, rejects no-common-type binaries, non-coercible
  operands for `! && || ^^`, an if condition not coercible to bool, non-numeric
  shift sides, bitwise on float, and return correctness (non-void must end in a
  return — recursing into a trailing block and a both-arms-return if/else; void
  can't return a value). Reads entries via resolved_entry_id but never mutates
  the symbol table or pushes frames.

Phase 4 — types beyond primitives
  * references (`T^`, no arithmetic)                                LANDED
    (kAddrOfExpr `^var` -> reference, kDerefExpr `v^` deref as lvalue+rvalue,
    kNullptrLiteral -> anyptr; `^` lexes as kBitXor, disambiguated structurally:
    prefix=address-of, postfix=deref [unless the next token starts a primary,
    then it's XOR], type-suffix=reference. kStoreStmt is the general lvalue-
    expression store [`ref^ = v`]. Compare-only `== !=` [v1 rule]; ordering and
    all arithmetic rejected. Both lower to LLVM `ptr`.)
  * iterators (`T[]`, additive arithmetic)                          LANDED
    (`iter^` deref, `iter[i]` subscript, `^arr[i]` -> iterator [operand-driven:
    var->`^`, indexed->`[]`]; `iter ± int` GEP by element stride, `iter += n` /
    `iter -= n` [classify iterator arm in kAugAssignStmt; desugars to
    `iter = iter + n`, other compound ops rejected "Arithmetic is not defined on
    a pointer."], `iter - iter` -> intptr [byte diff / sizeof elem], `++`/`--`
    step one element; full six comparisons. iterator->reference assignment demotes
    [v1]. `char[]` from a string literal.)
  * fixed-size arrays (`T name[N]`, multi-dim)                      LANDED
    (decl size after the NAME; multi-dim is STANDARD row-major — `int[5][3]` is 5
    chunks of 3 = LLVM `[5 x [3 x i32]]`, the LEFTMOST dim OUTERMOST; `arr[i]...`
    subscript [every dimension required] is one GEP with the source indices
    outer-to-inner [subscript[k] consumes dims[k]]; `^arr[i]` -> iterator;
    constant-index [incl. char] bounds-checked. The COMMA form `arr[a,b]` /
    `int g[3,5]` is the "natural order" transpose — a comma list desugars
    `[a,..,z] -> [z]...[a]` purely in the grammar [type-blind], so `g[x,y]` ==
    `g[y][x]` and `int[3,5]` == `int[5][3]`. OVER-indexing past the rank reports
    "Subscript indexes past the last dimension of '<arr>'"; a LITERAL zero/negative
    dim is rejected in the grammar [a const-EXPRESSION dim in constfold]. A dim may
    be a const EXPRESSION, not just a literal [`T a[N]`, `T a[sizeof(int)]`,
    `T a[N+1]`], in ANY type position [LANDED everywhere, not just the name]:
    parseType takes a dim_sink passed by every composition context — tuple slot,
    var-decl / param / return / alias-RHS / sizeof / cast / conversion target /
    for-var [a tuple slot is how a const dim rides a pointer/for-var/return type;
    `int[N]^` array-then-`^` isn't a type]. A non-literal dim spells a provisional
    `[1]` + pushes its expr; constfold's bakeDimsWalk folds + bakes PRE-ORDER over
    the whole TypeRef tree [tuple slots + array elements, through ptr/iter/alias],
    within the fixpoint so a later sizeof sees the real size; the final pass errors a
    non-constant / non-positive / non-integer dim. An ALIAS target's uses, expanded
    eagerly by resolve with the provisional `[1]`, are re-pointed by an AliasRefresh
    bakeNodeDims records [node types at walk-entry, entry types on bake]. The for-var
    head accepts a tuple type [looksLikeTupleTypeDecl] so a for-var carries one.
    [Array and tuple conversion targets land via the lockstep
    emitConvertWalk; the const-dim plumbing rides the same dim_sink path into
    the conversion target.]

    takes a NESTED tuple whose SHAPE matches the declared dims [row × col] —

    (kConvertExpr — the VALUE analog of the pointer cast: an explicit width/sign/
    family change that makes the implicit-coercion rejections legal, the operation
    every "use an explicit type conversion" diagnostic points at. It CHANGES the
    bits [a cast reinterprets them]; modeled as assignment-to-a-temp, so the
