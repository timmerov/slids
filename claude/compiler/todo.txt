BUGS

  * this should be a compile error:
    void fn(int a) { }
    void fn(int a, int b = 0) { }
    what does fn(1) mean?
    1. it's ambiguous, compile error. we can never call fn(a).
    2. it matches fn(int). the only way to call fn(a,b) is to pass two arguments.
    which means we can never use the default value of b.
    DECIDED (2026-07-10): option 1. CURRENT behavior — fn(1) is "Ambiguous call" (both
    viable at arity 1, equal exact rung; canon amb_one). TO BECOME a DECLARATION-time
    error (a default whose arity range overlaps another overload's fixed arity) in a
    future session; until then callers assume the pattern doesn't occur.

  * this should compile but doesn't:
    Shift(int a) { op<<(Shift^ lhs, Shift^ rhs) { a = lhs^.a << rhs^.a; } }
    (Shift, Shift) tuple = (Shift(1), Shift(3)) << 1;

  * ARENA REALLOC DANGLES `get()` REFERENCES (structural) — `widen::get(ref)`
    returns a `Type const&` into `arena().types`, a `std::vector<Type>`. Any intern
    (`intern` / `internArray` / `internTuple` / `internPointer` / `internIterator`,
    and `deepStrip`, which calls them) funnels through one `push_back` (widen.cpp:734)
    that can REALLOCATE the vector and dangle every outstanding `get()` reference.
    The whole codebase relies on an unwritten "capture-before-intern" discipline:
    read the fields you need into locals BEFORE calling an intern, never touch the
    `Type const&` after. Three sites violated it (read a dim count after interning a
    sub-array type) and were fixed per-site (classify::destructureSlots, plus the
    sweep fixes in emitConstructHooks / emitDestructHooks / the for-over-iterable
    slot expansion) — but the hazard recurs every time someone holds a `get()` ref
    across an intern, and only bites multi-dim arrays (the one case where the element
    type is freshly interned), so it slips tests. DURABLE FIX = make handles stable:
    back the arena with a node-stable container (`std::deque<Type>`, or a deque of
    chunks) so a `push_back` never moves existing elements, OR have `get()` return by
    value. Either removes the discipline requirement and the entire bug class. Until
    then, every `Type const& x = get(...)` near an intern is a latent dangling read.

  * TUPLE/ARRAY-OF-CLASS INLINE LITERAL — extra copy + late temp destruction.
    Initializing an aggregate of class VALUES from an inline literal —
    `(Tracked, Tracked) t = (Tracked(1), Tracked(2));` — materializes each element as a
    temporary and then COPY-constructs it into the destination slot (no RVO into the
    aggregate: 4 ctors where 2 would do), AND the element temporaries are destroyed at
    BLOCK exit rather than at the initializing statement's seam. It is BALANCED (every
    ctor has a matching dtor — no leak / double-free), so this is a lifetime / efficiency
    bug, not a crash. Contrast a FUNCTION-returned tuple-of-class (sret), which move-
    constructs into the slot and destroys the source husks promptly — the clean shape.
    Root is in emitExpr's tuple-literal-of-class lowering, NOT the declarator construct
    funnel. Observed while adding the destructure-into-class-slots coverage test
    (tuple/destructure.sl), which sidesteps it by using a tuple-returning function as the
    destructure source.

  * AGGREGATE-CONVERSION SOURCE RE-EVALUATED PER SLOT (side-effecting bare-lvalue
    access path). The aggregate-conversion spill (`lowerAggregateConversion`) and the
    class-field aggregate spread treat any BARE LVALUE source (`isBareLvalue`:
    ident/field/index/deref) as "re-index in place, no side effect to duplicate" — but
    an index/field whose ACCESS PATH has a side effect (`(Agg = arr[pick()])`,
    `(Agg = get()^.f)`) is CLONED per slot, so the side effect runs N times instead of
    once. Balanced/correct-valued, just a duplicated effect. Only a CALL/op source is
    spilled-once today; a lvalue with a side-effecting subscript/deref/base is not.
    Pre-existing for kIndexExpr/kDerefExpr; kFieldExpr joined the set with the
    isBareLvalue dedup (2026-07-08). FIX = spill any source whose access path is not
    provably side-effect-free (only a bare-ident chain is), not just non-lvalues.
    Surfaced reviewing the return_fn elide work; `sp = ((Amt,Amt) = mkpair())` (call
    source) already asserts spill-once, but no bare-lvalue-with-side-effect test exists.

  * NESTED-FUNCTION BODIES SKIP THE DESUGAR LOWERING PASSES — desugar's per-function
    loops (liftSretCallList, lowerAggregateList, lowerStatementList/PPID, analyzeNrvo)
    iterate only the TOP-LEVEL program functions (`program->children`); a NESTED function
    is a statement inside a host body, and liftSretCallList has no kFunctionDef case, so
    it is passed through un-recursed. Its body therefore never gets lift / aggregate-copy /
    PPID lowering. Result: inside a nested function, `++`/`--` ("inc/dec survived desugar's
    PPID pass"), a cross-form/leaf-widen aggregate COPY, and every CLASS TEMP (a
    `Class(args)` construction, a hook-returning call, a scalar OR aggregate value
    conversion `(Class = src)` / `((Class,int) = t)`) reach codegen unlowered and hit the
    `kConvertExpr needs 1 operand` / inc-dec asserts. Codegen already lifts nested fns to
    top-level LLVM (collectNestedFunctions), but the AST passes run before that and miss
    them. FIX = have each of the four desugar pass-loops process nested-function bodies
    too, each with ITS OWN return type (collect all fn defs recursively, or add a
    kFunctionDef-recurse to the list walkers). Pre-existing (class value conversions to a
    scalar class already crash in a nested fn); surfaced while landing aggregate class-leaf
    conversions — the canon neg_class_in_slot helper is written as a nested function.

  * CLASS-VALUED RHS ARG TEMP LEAKS TO ENCLOSING SCOPE — the seq-wrap that gives a
    nested arg / receiver temp STATEMENT lifetime in a var-decl / assign / return rhs
    (liftSretCallList, 2026-07-11) is SCALAR-VALUED-rhs only: when the rhs value is a
    CLASS built in place (`Class y = make(Class(80));`), its arg temp still registers its
    dtor in the ENCLOSING scope instead of dying at the statement. The class-valued rhs
    takes the in-place sret fast paths (kVarDeclStmt sret-intercept / kReturnStmt sret /
    kAssignStmt case-2/3), which need the RAW rhs node — a seq around it would defeat
    in-place and force an extra copy. BALANCED (no leak/double-free), a lifetime bug
    only. Pinned by test/class/evaluate.sl case 7. FIX = scope only the lifted PRE temps
    (not the in-place target) — destroy them right after the sret call completes, rather
    than wrapping the rhs value in a seq.

  * ● Tuple op= (t = s where the slots are classes with a user op=):
    - Today: the whole tuple is copied as one blob — load {{i32},{i32}} then store — a raw memory blit.
    - So each slot's op= (and even the default @Elem__$copy) is skipped. Elem::op= fired never runs.
    - Fine for trivial slots (the bytes are the value). Wrong for a slot whose copy is non-trivial — a heap-owning class
    blits its pointer shallow → two owners → double-free.
    - Deferred fix: lower a same-type tuple = per slot (t[i].op=(s[i]) / @Elem__$copy) instead of one blit, so each
    element's copy semantics run.

  deferred features involving classes:

  * CONSTRUCTION-IN-EXPRESSION — remaining gaps (record:
    [[project_self_and_method_calls]]):

      `r^ = Class()`, `w <-- Class()` are still rejected ("Constructing a class in
      this position") — those positions aren't lifted (see nameless.sl negatives).

  * CLASS-FIELD INIT FROM A SIZE-1 TUPLE — the "vexing parse" (Phase 5). CONTEXT,
    the class structure: a class IS a named tuple. `Name(field-list){body}` interns
    a kSlid that CARRIES its field slot types (the layout rides on the type, so
    codegen needs no symbol table), a ClassInfo holds the field names/defaults
    (parse-side only), `.field` lowers to a slot index, and construction reuses the
    tuple aggregate path. Two init forms feed it: the CALL form
    `Name(arg0, arg1, ...)` — each arg is ONE field's value, so a tuple arg stays
    WHOLE and a class-typed field constructs from it recursively (its own defaults
    fill the rest) — and the `=` form `Name x = tuple` — the init tuple is the
    FIELD tuple, SPREAD slot-to-field. They coincide for a multi-field class
    (`= (6,7)` == `(6, 7)`) and DIVERGE when one argument is itself a tuple bound
    for a single (class) field: the call form keeps it whole, `=` spreads.
    THE GAP: a SIZE-1 init tuple is INEXPRESSIBLE in the `=` form — `( X )` is a
    grouping paren that COLLAPSES to `X` (size-1 tuples collapse to their slot
    everywhere). So `Outer x = ( (13,17) )` (Outer has one class field) collapses
    to `= (13,17)`, a 2-slot tuple spread into 1 field -> "1 field(s) but 2
    initializer(s)". No `=`-spelling passes a whole tuple to a lone field; the
    CALL form `Outer x( (13,17) )` is the only way (parseCallArgs preserves the
    argument boundary the collapsing parens destroy). PUNTED: use the call /
    initializer-list form for a single-field-from-tuple init; canon initialize.sl
    keeps the working call form `Outer ot2( (13,17) )` and the `= ( (13,17) )` case
    commented with a "vexing parse" note. A real fix
    needs a non-collapsing 1-tuple syntax, which touches grouping parens globally —
    not worth it. prerequisite: a 1-tuple spelling (none planned).

  * x++ AS AN LVALUE (but NOT ++x) — the PPID model lifts the bump off the lvalue,
    so a POST-inc target is well-defined: `x++ = 42` -> `x = 42; x++` (store, THEN

    `++x = 42` would bump THEN store (`x = x+1; x = 42`), so the pre-bump is dead.
    So `x++` may be an lvalue; `++x` may not. This also makes chained `x++ ++` well
    defined (two bumps of x — see CHAIN POSTFIX PPID). Today parseIncDecStmt
    (grammar.cpp:1467) commits to the bare `x++;` statement the moment it sees
    `ident ++` and demands `;`, so the lvalue form never parses. Needs the
    statement/lvalue parse to accept a postfix-inc lvalue + desugar to lift the bump
    after the store.

REACH GOALS

  * inferred reference type.
    int x = 100;
    ref^ = x;
    type of ref is int^.
    actual use case:
    for (ref^ : container) { }
    the loop type is inferred from container.
    but we specify we want to loop by reference, not value.

  * b = (obj < x) desugars to b = obj.op<(x) regardless of x.
    if x is not a class or obj.op<(x) is not defined then we could try transposing
    the comparison:
    b = (x < obj)  -->  b = (obj > x)  -->  b = obj.op>(x)
    it's unlikely we'd want to apply this to all binary operations.

  * decide if we want to revisit expanding op==.
    Vector<int> vecint1; Vector<int> vecint2; Vector<bool> vecbool;
    vecbool = (vecint1 == vecint2);
    what does that map to?
    vecbool = vecint1.op==(vecint2);
    vecbool.op==(vecint1, vecint2);
    first case means we relax the restriction on what op== can return.
    the second case means we allow binary op==.

  * NESTED FUNCTION RESTRICTIONS TO REVISIT — three deliberate limits, each a
    compile error today (canon nested.sl says "needs to be revisited"):
      - DEEP NESTING — a nested function may not contain a further nested
        function ("Nested functions may not contain further nested functions.").
      - SIBLING CALLS — one nested function may not call another ("Calling a
        sibling nested function is not supported."). Lambda-lift threads only the
        host's captures, not a sibling's capture environment.
      - NESTED OVERLOADING — nested functions do NOT overload; a same-name nested
        pair is a "Duplicate declaration". classifyCall restricts candidates to
        file scope (parent_frame_id == 0). Revisit if nested overloads are wanted.

  * DROP THE ENUM QUALIFIER IN CONTEXT — when a variable's declared type is an
    enum, a bare member name on the rhs should resolve against that enum's
    namespace: `Language best = kRust;` then `best = kSlids;` (an implied
    `alias Language;` scoped to the statement / variable). Today bare `kRust` is
    "needs a namespace qualifier"; the canon block in enum.sl is commented
    pending this. Needs type-directed name resolution (the lvalue/assignment
    target type steering the rhs lookup) — no other feature needs it yet.
    prerequisite: lvalues.

  * ACCESS A SHADOWED LOCAL CLASS — a class defined in a function body may be
    shadowed by a same-named class in a nested function (allowed; bare lookup is
    innermost-first, so the inner one wins and the outer becomes unreachable — see
    nested_cls.sl). There is no syntax to name the shadowed (enclosing-scope) one.
    `::` is NOT it: `::` starts resolution at GLOBAL scope, but a local class lives
    in its function's body frame, not file scope, so `::` overshoots it. Nor can
    `::main:Function` work — `main` is a function, not a namespace, so it can't be
    a qualifier segment. The meaning of `::` is fixed (global) and will not change.
    Reaching an outer shadowed local class would need a DISTINCT "enclosing scope"
    qualifier, with its multi-level semantics (skip-one vs nearest-match-outward)
    still to be decided. No current need; the shadowed class is simply accepted as
    unreachable. prerequisite: local classes in function bodies.

  * a function returning a discarded array/tuple/class should be able to optimize.
    the caller can pass nullptr for the result slot. function must check.
    does not apply when the returned class has a ctor/dtor.

  * don't pass ^self to methods that don't need it.
    tricky bit is a class declared in a .slh header file.
    safest is for every method to get ^self and maybe not use it.
    otherwise it needs to be decorated somehow.

  * VIRTUAL DEFAULT ARGUMENTS — STATIC vs DYNAMIC. A virtual method's default
    argument currently binds from the STATIC receiver type (the C++ rule): a
    call `dp^.f()` through a `Def^` fills the default off `Def::f`'s signature,
    then dispatches to the dynamic body — so a `Ddef` override with a DIFFERENT
    default has its default silently ignored through a base pointer. This is a
    compile-time call-site rewrite (`f()` -> `f(<static-default>)`) feeding
    normal dispatch; the runtime type isn't known when the default is filled.
    Locked into virtual.sl's golden (`f() = 10`, not 18). Reach goal: decide
    whether Slids should instead resolve the default from the DYNAMIC type
    (non-C++, needs the default threaded through dispatch, not the call site).
    Purely a semantics choice — today's behavior is deliberate, not a bug.

    allocation) at compile time — a new[]/delete[] mismatch shouldn't reach codegen.

    value-counter continuation across openings is the undecided part.)

  * CHAIN POSTFIX PPID — repeated postfix inc/dec, `x++ ++ ++` (and the lexer's
    greedy `x+++++` -> `x ++ ++ +`). WELL-DEFINED in the PPID model (the bump lifts
    off the lvalue, so `x++ ++` is two bumps of x — NOT the C `(x++)++`-needs-an-
    lvalue problem; slids doesn't evaluate-then-rebump). Blocked GRAMMATICALLY, not
    semantically: parsePostfix applies ONE postfix ++/-- and returns (grammar.cpp:
    818), so a second `++` is a parse error. Low priority / low utility. Prereq:
    `x++ AS AN LVALUE` (above) — same lift-the-bump-off-the-lvalue machinery. Needs
    parsePostfix to LOOP ++/--.

  * UNIT ARRAY WITH A NON-SCALAR ELEMENT — `(int,int) arr[1] = (3,4)` (a size-1
    array whose element is itself a tuple / array / class) can't be initialized: the
    1-tuple==scalar collapse erases the wrapper, so `(3,4)` reads as TWO elements,
    not one tuple element -> "shape does not match". (The SCALAR-element case
    `int arr[1] = 2` / `int m[1][1] = 7` now works — checkValueAssign wraps the bare
    scalar in dims-deep 1-tuples.) Needs a non-collapsing 1-tuple spelling — same
    root as CLASS-FIELD INIT FROM A SIZE-1 TUPLE below; defer with it.

  * allow partial initialization of an array/tuple of class elements.
    Class arr[3] = (1,2); (Class,Class,Class) tup = (1,2);

  * REDUNDANT / NESTED TYPE GROUPING accepted silently — `((((int)))) x` parses as
    `int`, at any nesting depth. Each `(T)` layer is a size-1 tuple that collapses to
    grouping (the comma is the tuple marker; there is no 1-tuple), so the parens
    evaporate unless one scopes a qualifier (`(const int)^`). A bare `(int) x` is thus
    pointless parens. Open: reject / warn on grouping with nothing to scope. Note
    parseType recurses one frame per `(` layer, so a pathological depth is a theoretical
    parser-stack DoS. Same 1-tuple-collapse root as UNIT ARRAY WITH A NON-SCALAR ELEMENT
    above. Pinned by vardecl.sl (`((((int)))) grp`). Low priority; harmless today.
