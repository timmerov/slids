BUGS

  * SWAP WITH AN INC/DEC OPERAND — `x++ <--> y++` (and `--`) now parses, but the swap
    rejects it with "a swap operand must be an lvalue": the operand is the incremented
    form, not a plain one. Store / assignment targets already handle this — the bump lifts
    to the phrase edge and the residual lvalue is what's written — so the fix is to give
    both swap operands that same target lowering. Small, parallels the landed store path.


UNBLOCKED FEATURES

  * should we be able to take the address of a temporary variable?

  * EXPRESSION-TEMP ELIMINATION — PING-PONG, THE LAST ELIDE.
    a chain costs the minimum temporaries and a survivor dies at the semicolon.
    - PING-PONG: a class with a 2-arg `op+` but NO `op+=` cannot fuse (`acc.op+(acc, c)`
      would read the accumulator while writing it), so every un-fusable operand starts a
      FRESH BUFFER — one extra object per step. Two ALTERNATING buffers would need none of
      them. Pinned by evaluate.sl block N (class Buf); the buffers are correctly
      statement-scoped, so this is efficiency only.

  * SWAP-INIT ON A PRIMITIVE IS REJECTED. `int b <--> a;` errors "Swap-initialization
    requires a class type". Swap-init default-constructs the target and then re-dispatches
    as an ordinary swap statement, and only the construct half is class-only.

  * SIZEOF ACROSS TUs — `sizeof` is an internal CONSTANT today (the GEP-null/ptrtoint
    `__$sizeof` helper, folded per TU), which is correct while every importing TU can
    see the whole layout. For an INCOMPLETE class it can't, so sizeof must become an
    external FUNCTION, synthesized by the sibling like the other synthesized members.
    Deferred until incomplete classes cross a TU — the two land together or not at all.


BLOCKED FEATURES

  deferred features involving classes:

  * DUP-DEF / DUP-DECL NOTE CARET POSITION — the "first defined here" note carets
    the first DEFINITION (Entry.def_tok) and "first declared here" the first
    DECLARATION (Entry.tok); verified by manual probe only. The //-EXPECT-ERROR
    runner matches a message SUBSTRING, not caret line/col, so this attribution
    can't be locked by a .sl negative. Needs a caret-aware assertion the harness
    doesn't have.

  blocked by const correctness

  * NON-SCALAR CONST AT FILE SCOPE — a file-scope `const` of array / tuple /
    class / pointer type is rejected ("requires global storage, which is not yet
    supported", resolve.cpp:5127), but a non-scalar `global` of the same type now
    compiles via the compound-lazy path (desugar.cpp:2532). A non-scalar const IS
    an immutable global; route it through the same storage instead of rejecting.
    The const + global file-scope registration arms already share addVarEntry;
    the remaining work is routing the const arm's non-scalar case through the
    global-storage path rather than erroring. NOTE: such a const then inherits the
    global constant-initializer rule for free (classify's isConstantInit) — which is
    no constraint at all on a const, whose initializer is constant by definition.

  * is this overload allowed?
    void fn(mutable int^ p);
    void fn(const int^ p);
    void fn((const int)^ p);
    if it is then overload matching needs an update.
    TODAY: overload matching ranks mutable->const as an EXACT match (recursively, down to
    the leaf — see argConvertCost / overload_fn.sl), so all three would be MUTUALLY
    AMBIGUOUS. The update is either to REJECT such an overload set outright, or to add a
    tiebreak that lets const distinguish them.
    WHERE A REJECTION WOULD GO: checkOverloadDefaultCollisions (classify) already rejects
    an undecidable overload SET at its declaration — today only when default parameters
    make two candidates match the same call. A const tie is the same shape of error, so it
    belongs in that pass. Note nothing fires there today: the three parameter types are
    distinct type handles, so the collision is visible only to the RANKER.
    Gated on const enforcement (Phase 6) — decide the policy with that work.

  longer term features:

  * CONST ENFORCEMENT (Phase 6) — const representation/syntax is in (qualifier,
    param munge, `mutable`, the casts); ENFORCEMENT is not. const is unchecked
    today, so a const->mutable drop is silently accepted. To do:
      - lvalue carries const / rvalue read is mutable (the value-category model);
      - const->mutable rejected at assign + call sites without an explicit `<mutable>`;
      - no write through a const reference (every assign-family site);
      - const-overload matcher (match-key strips const);
      - method-level const + a const receiver `_$recv`;
      - `mutable` REQUIRED on a move/swap operator param (`op<--`/`op<-->`) — the
        operators parse + dispatch now; only the mutable-on-the-param REQUIREMENT
        (enforcement) waits on this const-enforcement work.
    cast.sl's const-correctness lines + the deep-const local forms in mutable.sl
    stay syntax-only until this lands.

  * NAMED-CONST-TRUE LOOP NOT FLAGGED — the constant-true-loop special case
    (3B revisited: a constant-true loop with no break is non-completing — its
    after-code is unreachable + it is a return-terminator) is detected
    SYNTACTICALLY in resolve (a bool/int/uint/char literal condition, incl. the
    synthesized empty `()`), because resolve runs before constfold. A loop whose
    condition is a NAMED const that folds to true (`const t = true; while (t)`) is
    therefore not flagged — it keeps the possibly-completing behavior (no
    unreachable-after, a trailing return is still demanded).
    NOT a small cleanup (investigated 2026): a safe fix needs BOTH halves, both
    in classify (post-constfold, where constTruth sees the folded named const):
      - set Node.non_completing for a const-true-no-break loop (so endsInReturnNode
        stops the trailing-return over-demand AND codegen emits the `unreachable`
        exit — codegen already reads non_completing, propagated via desugar).
      - a NEW classify-side unreachable-after pass: flag a statement following a
        non-completing loop "Unreachable statement." Without it, codegen's block
        loop emits ALL children, so code after the loop lands after the
        `unreachable` -> INVALID IR. (resolve's 2A does this syntactically; classify
        must mirror it value-aware. Only fires for the named-const case — the
        syntactic case already errored in resolve and never reaches classify.)
    Plus a minor wart: resolve's possibly-zero DA may report "use of uninitialized"
    on the (now-unreachable) after-code before classify flags it. Low value
    (`while (true)` / empty-cond are the common forms and fully handled), real cost.

  * TEMPLATED ALIAS — `alias X<...> = T;`.

  * UNUSED PARAMETER AS A COMPILE ERROR — NEEDS REVIEW. Today the unused-local
    sweep (resolve) covers only body-declared locals; parameters are exempt
    (vardecl.sl's `unused_param` compiles). Proposal: flag a parameter that is
    never read, like an unused local. NEEDS REVIEW — params are often legitimately
    unused (interface conformance, overload / signature symmetry, a callback that
    ignores an arg), so this likely wants an opt-out (a name convention or an
    explicit discard) rather than a blanket error. Decide the policy first.

  * TYPELESS CONST FROM A RUNTIME SCALAR — `int y = 42; const x = y;` errors
    "Initializer for 'x' is not a constant expression." x can't be a substituted
    literal constant (y is a runtime value), but it SHOULD be a const-qualified
    scalar VARIABLE (allocated + initialized, not-mutable) — like the typeless
    AGGREGATE consts that now route to variables (classify infers the rhs type and
    flips a non-primitive typeless const to a kLocalVar + deepConst). Extend that to
    a PRIMITIVE rhs that didn't fold: route to a const SCALAR variable instead of
    erroring. Keep a FOLDABLE scalar (`const x = 5`) on the substitution path.

  * OPENED-SCOPE NAME AMBIGUITY (not unique to inheritance — applies to ANY opened
    scope). resolveNameDetail flags a name found in two OPENED frames ("Reference to
    'X' is ambiguous"). the live residue is
    STATICS: a derived `const/alias/enum` with the SAME NAME as a base one is flagged
    ambiguous BARE — `inheritance.sl` pins this as a negative. WORKAROUND IN PLACE:
    qualify it (`Sa:kBase` / `Sd:kBase`), which `tryResolveBaseQualifier` now defers to
    the normal qualified-static lookup. Same tie also from `alias Ns1; alias Ns2;`
    sharing a member. STILL TO DECIDE: precedence of an opened base (or inner derived)
    vs a co-equal import — strict precedence (inner/derived shadows, no error) vs
    co-equal (ambiguous error). Lean: strict precedence, so "ambiguous" stays sourced
    only from multiple co-equal imports, and a derived static shadows its base's.

  * QUALIFIED-NAME COLLISIONS — DEFER ENTIRELY until a TRULY UNRESOLVABLE example
    surfaces. No case that normal
    scoping can't resolve deterministically has surfaced; revisit only when one does.


REACH GOALS

  * inferred reference type.
    int x = 100;
    ref^ = x;
    type of ref is int^.
    actual use case:
    for (ref^ : container) { }
    the loop type is inferred from container.
    but we specify we want to loop by reference, not value.

  * b = (obj < x) desugars to b = obj.op<(x) regardless of x.
    if x is not a class or obj.op<(x) is not defined then we could try transposing
    the comparison:
    b = (x < obj)  -->  b = (obj > x)  -->  b = obj.op>(x)
    it's unlikely we'd want to apply this to all binary operations.

  * decide if we want to revisit expanding op==.
    Vector<int> vecint1; Vector<int> vecint2; Vector<bool> vecbool;
    vecbool = (vecint1 == vecint2);
    what does that map to?
    vecbool = vecint1.op==(vecint2);
    vecbool.op==(vecint1, vecint2);
    first case means we relax the restriction on what op== can return.
    the second case means we allow binary op==.

  * NESTED FUNCTION RESTRICTIONS TO REVISIT — the remaining deliberate limits, each a
    compile error today (canon nested_fn.sl says "needs to be revisited"):
      - DEEP NESTING — a nested function may not contain a further nested
        function ("Nested functions may not contain further nested functions.").
        The check is now "am I inside a nested function's body" (capture_floor >= 0),
        so it catches an inner function hidden in a BLOCK too.
      - SIBLING CALLS — one nested function may not call another ("Calling a
        sibling nested function is not supported."). Lambda-lift threads only the
        host's captures, not a sibling's capture environment.
      - NESTED OVERLOADING — nested functions do NOT overload; a same-name nested
        pair is a "Duplicate declaration". classifyCall restricts candidates to
        file scope (parent_frame_id == 0). Revisit if nested overloads are wanted.

  * DROP THE ENUM QUALIFIER IN CONTEXT — when a variable's declared type is an
    enum, a bare member name on the rhs should resolve against that enum's
    namespace: `Language best = kRust;` then `best = kSlids;` (an implied
    `alias Language;` scoped to the statement / variable). Today bare `kRust` is
    "needs a namespace qualifier"; the canon block in enum.sl is commented
    pending this. Needs type-directed name resolution (the lvalue/assignment
    target type steering the rhs lookup) — no other feature needs it yet.
    prerequisite: lvalues.

  * ACCESS A SHADOWED LOCAL CLASS — a class defined in a function body may be
    shadowed by a same-named class in a nested function (allowed; bare lookup is
    innermost-first, so the inner one wins and the outer becomes unreachable — see
    nested_cls.sl). There is no syntax to name the shadowed (enclosing-scope) one.
    `::` is NOT it: `::` starts resolution at GLOBAL scope, but a local class lives
    in its function's body frame, not file scope, so `::` overshoots it. Nor can
    `::main:Function` work — `main` is a function, not a namespace, so it can't be
    a qualifier segment. The meaning of `::` is fixed (global) and will not change.
    Reaching an outer shadowed local class would need a DISTINCT "enclosing scope"
    qualifier, with its multi-level semantics (skip-one vs nearest-match-outward)
    still to be decided. No current need; the shadowed class is simply accepted as
    unreachable. prerequisite: local classes in function bodies.

  * a function returning a discarded array/tuple/class should be able to optimize.
    the caller can pass nullptr for the result slot. function must check.
    does not apply when the returned class has a ctor/dtor.

  * don't pass ^self to methods that don't need it.
    tricky bit is a class declared in a .slh header file.
    safest is for every method to get ^self and maybe not use it.
    otherwise it needs to be decorated somehow.

  * VIRTUAL DEFAULT ARGUMENTS — STATIC vs DYNAMIC. A virtual method's default
    argument currently binds from the STATIC receiver type (the C++ rule): a
    call `dp^.f()` through a `Def^` fills the default off `Def::f`'s signature,
    then dispatches to the dynamic body — so a `Ddef` override with a DIFFERENT
    default has its default silently ignored through a base pointer. This is a
    compile-time call-site rewrite (`f()` -> `f(<static-default>)`) feeding
    normal dispatch; the runtime type isn't known when the default is filled.
    Locked into virtual.sl's golden (`f() = 10`, not 18). Reach goal: decide
    whether Slids should instead resolve the default from the DYNAMIC type
    (non-C++, needs the default threaded through dispatch, not the call site).
    Purely a semantics choice — today's behavior is deliberate, not a bug.

ASPIRATIONAL FEATURES (someday; low priority, may never land)

  * this is a footgun. c++ has a similar issue. it's a bit easier with slids.
    ARRAY-NEW INTO A SINGLE REF, THEN SINGLE DELETE — `CtorDtor^ p = new CtorDtor[3];
    delete p;` COMPILES (iterator->ref demotion is allowed) but CRASHES at runtime
    (`munmap: invalid pointer`): the array alloc carries the 8-byte count cookie and
    `delete` of a single ref frees the returned ptr, not ptr-8, and runs no per-elem
    dtors. The correct `CtorDtor[] p = new CtorDtor[3]; delete p;` works (canon
    construct.sl). Fix: reject `T^ = new T[n]` (or a single delete of a cookie'd
    allocation) at compile time — a new[]/delete[] mismatch shouldn't reach codegen.

  * REOPEN ENUMS — re-`enum SameName ( ... )` to add members to an existing
    enum (member names can't collide across openings). Open spec qs: does the
    value counter restart at 0 on reopen, and must the first member of a
    reopening carry an explicit value? Defer until decided. (Namespaces already
    reopen; enums would reuse openNamespace's find-or-create, but the
    value-counter continuation across openings is the undecided part.)

  * CHAIN POSTFIX PPID — repeated postfix inc/dec, `x++ ++ ++` (and the lexer's
    greedy `x+++++` -> `x ++ ++ +`). WELL-DEFINED in the PPID model (the bump lifts
    off the lvalue, so `x++ ++` is two bumps of x — NOT the C `(x++)++`-needs-an-
    lvalue problem; slids doesn't evaluate-then-rebump). Blocked GRAMMATICALLY, not
    semantically: parsePostfix applies ONE postfix ++/-- and returns (grammar.cpp:
    818), so a second `++` is a parse error. Low priority / low utility. Prereq:
    `x++ AS AN LVALUE` (above) — same lift-the-bump-off-the-lvalue machinery. Needs
    parsePostfix to LOOP ++/--.

  * REDUNDANT / NESTED TYPE GROUPING accepted silently — `((((int)))) x` parses as
    `int`, at any nesting depth. Each `(T)` layer is a size-1 tuple that collapses to
    grouping (the comma is the tuple marker; there is no 1-tuple), so the parens
    evaporate unless one scopes a qualifier (`(const int)^`). A bare `(int) x` is thus
    pointless parens. Open: reject / warn on grouping with nothing to scope. Note
    parseType recurses one frame per `(` layer, so a pathological depth is a theoretical
    parser-stack DoS. Same 1-tuple-collapse root as UNIT ARRAY WITH A NON-SCALAR ELEMENT
    above. Pinned by vardecl.sl (`((((int)))) grp`). Low priority; harmless today.

  * review whether array to reference is a single cast or a chain of two casts.
    int arr[5]; ^arr[0] is iterator. fn(arr) desugars to fn(^arr[0]) so it can match
    fn(int[]). can fn(arr) also match fn(int^)? CURRENT IMPLEMENTATION: yes, ONE cast
    (argConvertCost element-decay = rung 2) — call.sl first_ref(rd) relies on it, and
    overload_fn.sl's no-match block was reworded to "single cast, subject to review".
    OPEN (user-deferred 2026-07-10): revisit whether it should be TWO casts (array ->
    int[] iterator, THEN int[] -> int^ iter->ref), which would REJECT fn(arr) -> fn(int^).

  * int x = x + 1; is clearly an error because x on the rhs is not initialized. but
    Class c = c + 1; c on the rhs *is* initialized. and therefore is a valid use.
    so we *could* allow it. probably don't want to.
    Class c += 1; would also be allowed.

  * the is-pod? question is too strict. all classes are not-pod. but a class with no
    ctor/dtor and no overloaded move/copy/swap operator is pod for that operation.
    STATE (2026-07-12): the transfer gate is now STRUCTURAL — `hasInPlaceClass`, "is a class
    physically in this storage?" — so EVERY class transfer calls `@Class__$copy` / `__$move`
    / `__$swap`, and a class can never be treated as POD for a transfer. That was the fix: the
    gate USED to be `typeNeedsHook` ("has a ctor body?"), which let a hook-less class fall
    into a blit twin that silently skipped the author's op<--. This item is the OPTIMIZATION
    on the other side of that fix: a class with no ctor/dtor AND only the SYNTHESIZED
    copy/move/swap has a transfer function whose body IS a memberwise blit, so the call could
    be inlined away at the site. Do it as a CODEGEN peephole keyed on "the class's op is
    synthesized and its fields are all POD" — never by weakening the gate, which is what
    reintroduces the bug. (LLVM already inlines these; measure before bothering.)

  * what should (a,b) = (b,a) do? IT SWAPS — the whole right side is evaluated BEFORE any
    target is written, so both slots read the OLD values. This is no longer an accident of
    how the source happened to be materialized: a destructure now binds its slots one at a
    time, so a literal that READS A TARGET is deliberately spilled to a temp first (an
    alias guard), precisely to keep the swap true. Pinned by tuple/destructure.sl.
    OPEN only as a SPEC question: confirm that is the wanted semantics. The alternative
    (store each slot as it is computed, so `(a,b) = (b,a)` leaves both equal to the old b)
    is what the per-slot lowering would do on its own, and it is cheaper by one temp.

  * revisit the decision to reject initialization of a class object field that requires a user
    defined move operator. Super s = (a, 1); s.a.op=(a) has to happen after s.a.ctor() and before
    s.ctor(). but there's no such seam. i'm not willing to inline ctors at the instantiation site.
    and i'm not willing to silently extract the assignment and perform it after default
    instantiation. Super s = (0,1); s.a.op=(a); let the user do that.
    claude speak:
    A CLASS FIELD TRANSFER IS NOT SUPPORTED. A field that would need a transfer — a same-class
    COPY (`Holder h( c );`), a same-class MOVE (`Holder h( mk() );`), or a value-init op= dispatch
    (`Super s = ((1,2,3),4);` where the field class has a matching user op=) — is currently
    REJECTED as a compile error rather than done (the landed mitigation; see readme-classes.txt
    "A CLASS FIELD THAT WOULD NEED A TRANSFER"). This item is the missing SUPPORT.
    WHY IT'S HARD: a construction's arguments are FIELD INITIALIZERS, so the enclosing CONSTRUCTOR
    MUST SEE THEM — a ctor body that computes `a_ + b_` reads the values passed, not the defaults.
    So the field's transfer cannot be hoisted past the enclosing ctor the way a tuple slot's can
    (a tuple has no ctor of its own — why that case was easy). It has to land BETWEEN the field's
    own ctor and the enclosing class's ctor body — the seam inside emitConstructed's hook recursion
    (fields' ctors, then this class's own ctor, then the vptr stamp), where the initializer
    expressions no longer exist. Pinned in the OTHER direction by evaluate.sl Q4 (class Hook):
    hoisting the field copies past the ctor makes Hook's ctor read 0 instead of 3.
    FIX = give construction a way to interleave: run the field hooks, then the field transfers,
    then the class's own ctor. Either codegen carries the field transfers into the construction,
    or the class's own-ctor call is separable so classify can emit decl(field hooks only) / field
    transfers / own-ctor as three steps. This is also why buildClassFromValue excludes a class
    FIELD (it field-lists, never op='s): the peel cannot hoist a field's transfer past the ctor.

  * allow empty slots in the `= (tuple)` VALUE-INIT form (`Class c = (,2,3)`). Two obstacles:
    it parses through the SHARED tuple-literal grammar, where an empty slot has no meaning outside
    a class value-init and the parser cannot know the target is a class (no syntactic gate like the
    construction `(` gives); and it classifies through buildClassFromValue's op= dispatch, a SECOND
    defaulting path separate from classifyClassInit's field loop. FIX: allow nullptr slots in the
    general tuple literal (every OTHER tuple consumer must then reject them), and default-fill the
    empties before the op= probe. Precedent is destructure's empty slots.

  * UNIT ARRAY WITH A NON-SCALAR ELEMENT — `(int,int) arr[1] = (3,4)` (a size-1
    array whose element is itself a tuple / array / class) can't be initialized: the
    1-tuple==scalar collapse erases the wrapper, so `(3,4)` reads as TWO elements,
    not one tuple element -> "shape does not match". (The SCALAR-element case
    `int arr[1] = 2` / `int m[1][1] = 7` now works — checkValueAssign wraps the bare
    scalar in dims-deep 1-tuples.) Needs a non-collapsing 1-tuple spelling, which
    touches grouping parens globally (none planned).
