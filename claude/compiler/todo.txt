BUGS

  * A CONSTRUCTION AS THE LEFT OPERAND OF A CLASS BINARY ONLY DISPATCHES IF THE CLASS HAS THE
    COMPOUND OPERATOR. `Add(7) + a` reports "Operator '+' is not defined on class 'Add'" for
    a class with a 2-arg `op+` and no `op+=`, while `mk() + a` (a CALL ‚Äî the same rvalue) and
    `a + Add(7)` (the same construction, on the right) both work. stampClassBinary's
    COLLAPSED-HEAD role treats a construction head as the accumulator itself, so a head built
    WITH args demands `op<OP>=` (op= would discard those args) and gives up when there is
    none. The collapse is an OPTIMIZATION (it saves a temp); with a 2-arg `op<OP>` available
    the construction should simply fall back to being a REAL OPERAND, materialized like any
    other rvalue. FIX = in the collapsed-head arm, `if (!viable && bin_id >= 0)` drop the
    collapse and take the real-operand path ‚Äî but desugar re-derives the role from the AST
    (lhs.kind == construction), so both ends have to agree; the stamp must carry the decision.
    This is what stops test/misc/bug.sl's line from compiling even with the aggregate explode.

  * ARENA REALLOC DANGLES `get()` REFERENCES (structural) ‚Äî `widen::get(ref)`
    returns a `Type const&` into `arena().types`, a `std::vector<Type>`. Any intern
    (`intern` / `internArray` / `internTuple` / `internPointer` / `internIterator`,
    and `deepStrip`, which calls them) funnels through one `push_back` (widen.cpp:734)
    that can REALLOCATE the vector and dangle every outstanding `get()` reference.
    The whole codebase relies on an unwritten "capture-before-intern" discipline:
    read the fields you need into locals BEFORE calling an intern, never touch the
    `Type const&` after. Three sites violated it (read a dim count after interning a
    sub-array type) and were fixed per-site (classify::destructureSlots, plus the
    sweep fixes in emitConstructHooks / emitDestructHooks / the for-over-iterable
    slot expansion) ‚Äî but the hazard recurs every time someone holds a `get()` ref
    across an intern, and only bites multi-dim arrays (the one case where the element
    type is freshly interned), so it slips tests. DURABLE FIX = make handles stable:
    back the arena with a node-stable container (`std::deque<Type>`, or a deque of
    chunks) so a `push_back` never moves existing elements, OR have `get()` return by
    value. Either removes the discipline requirement and the entire bug class. Until
    then, every `Type const& x = get(...)` near an intern is a latent dangling read.

  * A CLASS FIELD INITIALIZED FROM A CLASS LVALUE IS CONSTRUCTED OVER THE TOP OF THE COPY.
    Given a class C whose constructor writes its own field, `Holder h( c );` leaves h's
    field holding what C's CONSTRUCTOR wrote, not what was in `c`. A wrong answer.
    WHY: the field is FILLED from `c` and only THEN finalized (the ctor hooks run), so the
    field's own constructor lands on top of the copied value and overwrites it. This is the
    same defect a DECLARATION had ("a class can only be copied into" ‚Äî fixed: the transfer
    is peeled off the decl and re-emitted against the constructed object).
    IT CANNOT BE FIXED THE SAME WAY, and the reason is worth keeping: a construction's
    arguments are FIELD INITIALIZERS, so the enclosing CONSTRUCTOR MUST SEE THEM. A ctor
    body that computes `a_ + b_` has to read the values that were passed, not the defaults.
    So the field's copy cannot be hoisted past the enclosing ctor the way a tuple slot's
    can (a tuple has no ctor of its own, which is exactly why that case was easy). It has
    to land BETWEEN the field's own ctor and the enclosing class's ctor body.
    That seam is inside emitConstructed's hook recursion (fields' ctors, then this class's
    own ctor, then the vptr stamp), where the initializer expressions no longer exist.
    FIX = give construction a way to interleave: run the field hooks, then the field
    transfers, then the class's own ctor. Either codegen carries the field transfers into
    the construction, or the class's own-ctor call is separable so classify can emit
    decl(field hooks only) / field copies / own-ctor as three steps.
    Pinned in the OTHER direction by evaluate.sl Q4 (class Hook): hoisting the field copies
    past the ctor makes Hook's ctor read 0 instead of 3 ‚Äî so a naive fix here is caught.

  * AGGREGATE-CONVERSION SOURCE RE-EVALUATED PER SLOT (side-effecting bare-lvalue
    access path). The aggregate-conversion spill (`lowerAggregateConversion`) and the
    class-field aggregate spread treat any BARE LVALUE source (`isBareLvalue`:
    ident/field/index/deref) as "re-index in place, no side effect to duplicate" ‚Äî but
    an index/field whose ACCESS PATH has a side effect (`(Agg = arr[pick()])`,
    `(Agg = get()^.f)`) is CLONED per slot, so the side effect runs N times instead of
    once. Balanced/correct-valued, just a duplicated effect. Only a CALL/op source is
    spilled-once today; a lvalue with a side-effecting subscript/deref/base is not.
    FIX = spill any source whose access path is not provably side-effect-free, not just
    non-lvalues. Both halves already exist and just need wiring together: the PREDICATE is
    classify's `isReReadableLvalue` (the destructure asks the identical question ‚Äî it reads
    its source once PER SLOT ‚Äî and answers it correctly: an ident/field/deref/const-index
    chain is safe to re-read, anything else spills), and the SPILL is classify's
    `spillToTemp` funnel (which also makes the temp's lifetime an explicit choice rather than
    an accident of where the decl is parked). These sites should use both.
    The slot-wise explode is the THIRD site to ask the question and the second to answer it
    right (`isReReadableOperand` + `prepareOperand` + `spillToTemp`); the two sites named here
    are the ones still left over. Fold all of it into ONE take-the-source-apart helper.
    No bare-lvalue-with-side-effect test exists yet.


UNBLOCKED FEATURES

  * EXPRESSION-TEMP ELIMINATION ‚Äî PING-PONG, THE LAST ELIDE.
    a chain costs the minimum temporaries and a survivor dies at the semicolon.
    - PING-PONG: a class with a 2-arg `op+` but NO `op+=` cannot fuse (`acc.op+(acc, c)`
      would read the accumulator while writing it), so every un-fusable operand starts a
      FRESH BUFFER ‚Äî one extra object per step. Two ALTERNATING buffers would need none of
      them. Pinned by evaluate.sl block N (class Buf); the buffers are correctly
      statement-scoped, so this is efficiency only.

  * SWAP-INIT ON A PRIMITIVE IS REJECTED. `int b <--> a;` errors "Swap-initialization
    requires a class type". Swap-init default-constructs the target and then re-dispatches
    as an ordinary swap statement, and only the construct half is class-only.

  * A DECLARATION FROM A TUPLE IGNORES THE CLASS'S op=. Given a class with a matching
    `op=((int,int)^)`, `Pair p = (1, 2);` SILENTLY FIELD-INITS (p is 1,2 ‚Äî the operator
    never runs), while the assignment form `q = (3, 4);` on an existing object dispatches
    the operator correctly. A silent wrong answer, not a missing feature.
    WHY: grammar gives a `(args)` construction and a `= (tuple)` initializer the SAME shape
    (one tuple child), so the declaration cannot tell "construct from these field values"
    apart from "assign me this tuple", and the operator dispatch is turned off for a tuple
    child rather than guess.
    FIX: mark the two forms apart in the parser, then let a declaration from a tuple take
    the same op= dispatch an assignment does.

  * SHOULD A WEAK LITERAL BIND TO A CLASS PARAMETER? `Shift s = 1;` field-fills a class in a
    DECLARATION, but `1` does NOT convert to a `Shift^` parameter, so a class operator
    `op<<(Shift^, Shift^)` cannot be called as `s << 1`. Open question, deferred: either the
    operator is written `op<<(Shift^, int)` (works today), or an argument position gains the
    same literal-to-class fill a declaration has. Needed by test/misc/bug.sl's line.


BLOCKED FEATURES

  deferred features involving classes:

  * CLASS-FIELD INIT FROM A SIZE-1 TUPLE ‚Äî the "vexing parse" (Phase 5). CONTEXT,
    the class structure: a class IS a named tuple. `Name(field-list){body}` interns
    a kSlid that CARRIES its field slot types (the layout rides on the type, so
    codegen needs no symbol table), a ClassInfo holds the field names/defaults
    (parse-side only), `.field` lowers to a slot index, and construction reuses the
    tuple aggregate path. Two init forms feed it: the CALL form
    `Name(arg0, arg1, ...)` ‚Äî each arg is ONE field's value, so a tuple arg stays
    WHOLE and a class-typed field constructs from it recursively (its own defaults
    fill the rest) ‚Äî and the `=` form `Name x = tuple` ‚Äî the init tuple is the
    FIELD tuple, SPREAD slot-to-field. They coincide for a multi-field class
    (`= (6,7)` == `(6, 7)`) and DIVERGE when one argument is itself a tuple bound
    for a single (class) field: the call form keeps it whole, `=` spreads.
    THE GAP: a SIZE-1 init tuple is INEXPRESSIBLE in the `=` form ‚Äî `( X )` is a
    grouping paren that COLLAPSES to `X` (size-1 tuples collapse to their slot
    everywhere). So `Outer x = ( (13,17) )` (Outer has one class field) collapses
    to `= (13,17)`, a 2-slot tuple spread into 1 field -> "1 field(s) but 2
    initializer(s)". No `=`-spelling passes a whole tuple to a lone field; the
    CALL form `Outer x( (13,17) )` is the only way (parseCallArgs preserves the
    argument boundary the collapsing parens destroy). PUNTED: use the call /
    initializer-list form for a single-field-from-tuple init; canon initialize.sl
    keeps the working call form `Outer ot2( (13,17) )` and the `= ( (13,17) )` case
    commented with a "vexing parse" note. A real fix
    needs a non-collapsing 1-tuple syntax, which touches grouping parens globally ‚Äî
    not worth it. prerequisite: a 1-tuple spelling (none planned).

REACH GOALS

  * inferred reference type.
    int x = 100;
    ref^ = x;
    type of ref is int^.
    actual use case:
    for (ref^ : container) { }
    the loop type is inferred from container.
    but we specify we want to loop by reference, not value.

  * b = (obj < x) desugars to b = obj.op<(x) regardless of x.
    if x is not a class or obj.op<(x) is not defined then we could try transposing
    the comparison:
    b = (x < obj)  -->  b = (obj > x)  -->  b = obj.op>(x)
    it's unlikely we'd want to apply this to all binary operations.

  * decide if we want to revisit expanding op==.
    Vector<int> vecint1; Vector<int> vecint2; Vector<bool> vecbool;
    vecbool = (vecint1 == vecint2);
    what does that map to?
    vecbool = vecint1.op==(vecint2);
    vecbool.op==(vecint1, vecint2);
    first case means we relax the restriction on what op== can return.
    the second case means we allow binary op==.

  * NESTED FUNCTION RESTRICTIONS TO REVISIT ‚Äî the remaining deliberate limits, each a
    compile error today (canon nested_fn.sl says "needs to be revisited"):
      - DEEP NESTING ‚Äî a nested function may not contain a further nested
        function ("Nested functions may not contain further nested functions.").
        The check is now "am I inside a nested function's body" (capture_floor >= 0),
        so it catches an inner function hidden in a BLOCK too.
      - SIBLING CALLS ‚Äî one nested function may not call another ("Calling a
        sibling nested function is not supported."). Lambda-lift threads only the
        host's captures, not a sibling's capture environment.
      - NESTED OVERLOADING ‚Äî nested functions do NOT overload; a same-name nested
        pair is a "Duplicate declaration". classifyCall restricts candidates to
        file scope (parent_frame_id == 0). Revisit if nested overloads are wanted.

  * DROP THE ENUM QUALIFIER IN CONTEXT ‚Äî when a variable's declared type is an
    enum, a bare member name on the rhs should resolve against that enum's
    namespace: `Language best = kRust;` then `best = kSlids;` (an implied
    `alias Language;` scoped to the statement / variable). Today bare `kRust` is
    "needs a namespace qualifier"; the canon block in enum.sl is commented
    pending this. Needs type-directed name resolution (the lvalue/assignment
    target type steering the rhs lookup) ‚Äî no other feature needs it yet.
    prerequisite: lvalues.

  * ACCESS A SHADOWED LOCAL CLASS ‚Äî a class defined in a function body may be
    shadowed by a same-named class in a nested function (allowed; bare lookup is
    innermost-first, so the inner one wins and the outer becomes unreachable ‚Äî see
    nested_cls.sl). There is no syntax to name the shadowed (enclosing-scope) one.
    `::` is NOT it: `::` starts resolution at GLOBAL scope, but a local class lives
    in its function's body frame, not file scope, so `::` overshoots it. Nor can
    `::main:Function` work ‚Äî `main` is a function, not a namespace, so it can't be
    a qualifier segment. The meaning of `::` is fixed (global) and will not change.
    Reaching an outer shadowed local class would need a DISTINCT "enclosing scope"
    qualifier, with its multi-level semantics (skip-one vs nearest-match-outward)
    still to be decided. No current need; the shadowed class is simply accepted as
    unreachable. prerequisite: local classes in function bodies.

  * a function returning a discarded array/tuple/class should be able to optimize.
    the caller can pass nullptr for the result slot. function must check.
    does not apply when the returned class has a ctor/dtor.

  * don't pass ^self to methods that don't need it.
    tricky bit is a class declared in a .slh header file.
    safest is for every method to get ^self and maybe not use it.
    otherwise it needs to be decorated somehow.

  * VIRTUAL DEFAULT ARGUMENTS ‚Äî STATIC vs DYNAMIC. A virtual method's default
    argument currently binds from the STATIC receiver type (the C++ rule): a
    call `dp^.f()` through a `Def^` fills the default off `Def::f`'s signature,
    then dispatches to the dynamic body ‚Äî so a `Ddef` override with a DIFFERENT
    default has its default silently ignored through a base pointer. This is a
    compile-time call-site rewrite (`f()` -> `f(<static-default>)`) feeding
    normal dispatch; the runtime type isn't known when the default is filled.
    Locked into virtual.sl's golden (`f() = 10`, not 18). Reach goal: decide
    whether Slids should instead resolve the default from the DYNAMIC type
    (non-C++, needs the default threaded through dispatch, not the call site).
    Purely a semantics choice ‚Äî today's behavior is deliberate, not a bug.

  * REDUNDANT / NESTED TYPE GROUPING accepted silently ‚Äî `((((int)))) x` parses as
    `int`, at any nesting depth. Each `(T)` layer is a size-1 tuple that collapses to
    grouping (the comma is the tuple marker; there is no 1-tuple), so the parens
    evaporate unless one scopes a qualifier (`(const int)^`). A bare `(int) x` is thus
    pointless parens. Open: reject / warn on grouping with nothing to scope. Note
    parseType recurses one frame per `(` layer, so a pathological depth is a theoretical
    parser-stack DoS. Same 1-tuple-collapse root as UNIT ARRAY WITH A NON-SCALAR ELEMENT
    above. Pinned by vardecl.sl (`((((int)))) grp`). Low priority; harmless today.

  * int x = x + 1; is clearly an error because x on the rhs is not initialized. but
    Class c = c + 1; c on the rhs *is* initialized. and therefore is a valid use.
    so we *could* allow it. probably don't want to.
    Class c += 1; would also be allowed.

  * the is-pod? question is too strict. all classes are not-pod. but a class with no
    ctor/dtor and no overloaded move/copy/swap operator is pod for that operation.
    STATE (2026-07-12): the transfer gate is now STRUCTURAL ‚Äî `hasInPlaceClass`, "is a class
    physically in this storage?" ‚Äî so EVERY class transfer calls `@Class__$copy` / `__$move`
    / `__$swap`, and a class can never be treated as POD for a transfer. That was the fix: the
    gate USED to be `typeNeedsHook` ("has a ctor body?"), which let a hook-less class fall
    into a blit twin that silently skipped the author's op<--. This item is the OPTIMIZATION
    on the other side of that fix: a class with no ctor/dtor AND only the SYNTHESIZED
    copy/move/swap has a transfer function whose body IS a memberwise blit, so the call could
    be inlined away at the site. Do it as a CODEGEN peephole keyed on "the class's op is
    synthesized and its fields are all POD" ‚Äî never by weakening the gate, which is what
    reintroduces the bug. (LLVM already inlines these; measure before bothering.)

  * what should (a,b) = (b,a) do? IT SWAPS ‚Äî the whole right side is evaluated BEFORE any
    target is written, so both slots read the OLD values. This is no longer an accident of
    how the source happened to be materialized: a destructure now binds its slots one at a
    time, so a literal that READS A TARGET is deliberately spilled to a temp first (an
    alias guard), precisely to keep the swap true. Pinned by tuple/destructure.sl.
    OPEN only as a SPEC question: confirm that is the wanted semantics. The alternative
    (store each slot as it is computed, so `(a,b) = (b,a)` leaves both equal to the old b)
    is what the per-slot lowering would do on its own, and it is cheaper by one temp.
