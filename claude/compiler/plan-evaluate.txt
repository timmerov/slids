GOAL (met)
  A class-operator chain costs the MINIMUM number of temporary objects, and any temporary
  that does survive dies at the END OF THE STATEMENT, not at the end of the enclosing scope.

WHAT MOVED (test/class/exp.evaluate, the J block)
  BEFORE                                          AFTER
  Acc jd = ja + jb;      1 temp                   0 temps
  Acc je = ja + jb + jc; 2 temps                  0 temps
  jf = ja + jb + jc;     3 temps                  1 temp, dead at the semicolon
  12 ctor/dtor pairs, ALL leaking to block end    7 pairs

WHY THE OLD FUSE IS GONE (do not re-derive it)
  classify's tryLowerBinaryChain keyed the fuse on the ASSIGNMENT TARGET because it ran
  BEFORE the infer loop and had no operand types. That inverted operator precedence (the
  `=` chose the `+`) and re-associated the chain, clobbering the target mid-expression.
  Deleted 2026-07-11 along with flattenLeftChain / referencesEntry / the context-steering
  arm of tryLowerClassBinary. See plan.txt "BINARY SIGNATURE RULE".

  A binary dispatches on the LHS OPERAND's class and nothing else (its first parameter MUST
  be a reference to the enclosing class). That stayed true here. This design never lets a
  destination choose an OPERATOR; it only chooses where the already-decided accumulator LIVES.


------------------------------------------------------------------------------
DECISION 1 — WHERE THE ACCUMULATOR LIVES  (reuse the funnel's existing question)
------------------------------------------------------------------------------

The declarator funnel already draws exactly the line this optimization needs:

    IS THE DESTINATION RAW STORAGE BEING CONSTRUCTED, OR A LIVE OBJECT BEING ASSIGNED?

  RAW STORAGE of the chain's exact class -> the accumulator IS the destination. ZERO temps.
      a var-decl; a RETURN (the returned local NRVOs onto %sret.in, so the CALLER's storage
      is the accumulator).
  A LIVE OBJECT -> a temp accumulator, then a MOVE in. Wrapped in a BLOCK, so the temp dies
      at the semicolon.
      kAssignStmt, kStoreStmt (field / index / deref target).
  NO destination, or a TYPE MISMATCH -> a lifted `_$optmp`, statement-scoped by the existing
      liftSretCallList wrap.
      a call argument, a nested operand; and `Other s = String + a` (result class String,
      destination Other -> the binding funnel converts).

WHY A LIVE TARGET CAN NEVER BE THE ACCUMULATOR (canon 12-16 says temp+move; here is why):
  - Making it the accumulator means CONSTRUCTING it — running its ctor. You cannot
    re-construct a live object.
  - Even with a DEFAULT head it fails: the seed can FALL BACK to op+= (Decision 2), and
    `s.op+=(a)` on a live s accumulates onto s's OLD VALUE — silently wrong.
  - And `s = String + a + s` aliases.

WHY A FRESH DECL TARGET NEEDS NO GUARD:
  A variable cannot appear in its own initializer — `Acc c = c + 1;` already errors with
  "Use of uninitialized variable 'c'." (verified). So eliding into a fresh declaration is
  unconditionally safe. The only guard is the EXACT-TYPE check.


------------------------------------------------------------------------------
DECISION 2 — THE LADDER  (which operator each operand is applied with)
------------------------------------------------------------------------------

  1. A fresh CONSTRUCTION at the head — a bare `Class`, `Class()`, or `Class(args)` —
     COLLAPSES INTO the accumulator, constructed with those args. Never an operand.
     AMENDED 2026-07-13: the collapse is an OPTIMIZATION (it saves one temp), and rules 2/3
     are what it needs to be POSSIBLE — an operator that ADVANCES the accumulator with one
     argument. A class with only the 2-arg `op<OP>` has none, and the chain used to give up and
     report the operator UNDEFINED, though rule 4 could have run it. So rule 1 is DECLINED
     there: the construction becomes an ORDINARY OPERAND of rule 4, one temp, dead at the
     semicolon — exactly what the same construction on the RIGHT always cost. (A DEFAULT head
     always has an `op=` — the synthesized copy — so it always collapses; the decline only
     fires on a head built WITH ARGS.) The ROLE is the STAMP's, not a shape desugar re-derives
     — see DECISION 3.
  2. First application to a DEFAULT-constructed accumulator: prefer `acc.op=(rhs)`, fall back
     to `acc.op<OP>=(rhs)`. This is canon 4-10 (String defines op= + op+= and NO 2-arg op+).
     RATIONALE (the user's): the accumulator is ALWAYS a fully CONSTRUCTED object before any
     operator runs — the funnel guarantees it. So nothing about construction forces op=;
     op= is chosen because it is CHEAPER than op+= on an empty object.
     RECORDED CONSEQUENCE: the seed does NOT consult the head operator symbol.
     `Vec v = Vec - a - b` lowers to `v.op=(a); v -= b` -> `a - b` (not `-a - b`). Accepted.
  3. First application to an accumulator constructed WITH ARGUMENTS: `acc.op<OP>=(rhs)` ONLY.
     FORCED, not a preference: op= would DISCARD the construction arguments.
     `Acc jd = Acc(5) + ja` must be `jd(5); jd += ja` -> 5+ja, never `jd = ja`.
  4. Head is a real OPERAND PAIR: the 2-arg `op<OP>(x, y)` when defined — ONE call; else
     DECOMPOSE to rules 2+5: `acc.op=(x); acc.op<OP>=(y)`. (The decompose is motivated by the
     ABSENCE of a 2-arg operator, NOT by construction.)
  5. Every LATER operand: `acc.op<OP>=(operand)` — the fuse, with ITS OWN operator, so a
     mixed chain (`a + b - c`) really subtracts.
  6. An un-fusable continuation (a 2-arg op<OP> but NO op<OP>=): fusing would self-alias
     (`acc.op+(acc, c)` reads acc while writing it), so it starts a FRESH BUFFER. One extra
     object per un-fusable step; the accumulator the destination becomes is the LAST one.
     STILL DEFERRED — PING-PONG: two alternating buffers would need none of them.


------------------------------------------------------------------------------
DECISION 3 — WHICH STAGE  (classify STAMPS, desugar LOWERS)
------------------------------------------------------------------------------

Rejected: doing the fuse in classify. It would be CORRECT (bottom-up, operand-keyed, no
precedence risk — unlike the deleted target-keyed one). But it re-thickens the pass we just
thinned, and it leaves the ELIDE with nowhere to live: knowing whether the accumulator can
occupy jd's slot requires seeing the chain and its destination TOGETHER, and desugar is the
only pass that has the whole statement in one piece at lowering time.

  CLASSIFY  stampClassBinary (was tryLowerClassBinary). Same resolution (result class = the
    LHS OPERAND's class) but it RECORDS on the kBinaryExpr instead of rewriting it: five
    operator entry ids (2-arg op<OP>, op<OP>=, op=(lhs), op=(rhs), op<--(Self^)) + the result
    class's default field-init tuple as children[2]. Diagnostics STAY here: a class with no
    viable ladder returns false and the existing clean error fires. It reads its own ROLE off
    its lhs (continuation / collapsed head / real-operand head) and checks viability per role.
    THE ROLE IS THE STAMP'S TOO (`op_collapse_head`, 2026-07-13). Desugar used to RE-DERIVE the
    collapsed-head role from the AST (lhs.is_construction), which meant classify could not
    DECIDE it — and so could not DECLINE the collapse when the class had no operator to advance
    the accumulator with (rule 1, amended). Anything classify decides, classify must SAY: a
    decision re-derived downstream is a decision classify does not actually own.
  DESUGAR   expandOpChainStmt (destinations) + lowerOpChain (the ladder), running inside
    liftSretCallList so the generated op calls get their own nested rvalues lifted like
    hand-written ones. Precedent: tryDesugarAugAssign — classify stamps, desugar synthesizes.

  DEVIATIONS FROM THE ORIGINAL PLAN (all deliberate):
    - lowerClassOperatorTemp was NOT deleted: the arity-1 UNARY still uses it. Moving unary
      too doubles the surface for a case not in canon. Missed elide, not a regression.
      CLOSED 2026-07-13 — the unary IS on the chain road now (stampClassUnary + a unary head
      in lowerOpChain; the spine walk stops at a unary so `-(a+b)` cannot self-alias), and
      lowerClassOperatorTemp is deleted. It cost MORE than a missed elide: the `_$optmp` was
      COPIED into the destination (so a live target never ran the author's op<--) and it kept
      ENCLOSING-scope lifetime. Canon evaluate.sl block U (class Neg).
    - classify still BUILDS the accumulator's construction value (classifyClassInit parks the
      default field tuple as children[2]). Desugar cannot: default field values are classify's
      job. This keeps construction on the ONE funnel.
    - The MOVE into a live target is a CALL to the stamped op<--, not a bare move node. See
      below — this is the one that bit.


------------------------------------------------------------------------------
THE BUG THIS LANDING FOUND — THE TRANSFER GATES
------------------------------------------------------------------------------

The first cut emitted a bare kMoveStmt for the live-target move. It was WRONG, and the way
it was wrong is worth keeping:

  Every HAND-WRITTEN transfer (`a <-- b`) is dispatched in CLASSIFY (dispatchAssignInit
  rewrites it to a method call, and methodSymbol renames op<--(Self^) to @Class__$move). A
  transfer synthesized in DESUGAR never passes through that. It falls to codegen — where all
  three transfer sites (emitInitFill's move arm, its copy arm, kSwapStmt) gated the dispatch
  on `typeNeedsHook` ("does a ctor body exist?"). A class can answer NO to that. A hook-less
  class fell past the gate into an inline load/store TWIN that re-implemented the transfer and
  SILENTLY SKIPPED THE CLASS'S OPERATOR.

FIXED at both ends:
  - the chain's move is emitted as a CALL through the stamped op<-- entry id (same makeOpCall
    funnel as every other rung), so it names the operator and depends on no gate;
  - the three gates now ask the STRUCTURAL question — `widen::hasInPlaceClass` ("is a class
    physically in this storage?") — which a class can never answer no to. The blit twins are
    unreachable for a class-bearing type; only pure POD load/stores.

  A class transfer now has exactly ONE implementation: the operator body. THAT is the
  invariant, and it is why this stops being a periodic discovery.

  NOTE: `typeNeedsHook` is still RIGHT where it remains (emitConstructed, dtor registration,
  the sret hook decisions) — those really are asking "is there a ctor body to call?". The two
  predicates are NOT interchangeable. The bug was using the behavioral one for a structural
  question.

COUNTING LIVES IN evaluate.sl
------------------------------------------------------------------------------

operator.sl's OpDefs/Sum have no ctor/dtor, so a fuse and a no-fuse print identically there —
which is exactly how the old target-keyed chain fuse survived for months with green goldens.
Coverage classes — READ THIS TABLE BEFORE ADDING ONE. Each is a distinct OPERATOR SET, chosen
so that a question has exactly one class that can ask it; a new class that duplicates a set
buys nothing (Z8 was first written with a new `Bin` that was Buf character for character).
All of these PRINT:
  Acc   ctor/dtor + op= + op<-- + op+= + op-= + 2-arg op+   — the fuse and the mixed chain
  Str   ctor/dtor + op= + op<-- + op+=, NO 2-arg op+        — the only-op+= ladder (canon 51-56)
  Buf   ctor/dtor + op= + 2-arg op+, NO op+=                — the un-fusable buffer + lifetime,
        and (Z8) the DECLINED head collapse: with no op+= there is nothing to advance a
        collapsed accumulator with, so a construction head must become an ordinary operand
  Triv  NO ctor/dtor, printing op= / op<-- / op+=           — the transfer gates
  Sw    NO ctor/dtor, printing op= / op<-- / op<-->         — the SWAP gate + aggregates
  Trk   ctor/dtor + printing op= + op<-- + op<-->            — the tuple-of-classes counts
Three classes DON'T print, because their questions are not about a print order:
  Ord   ctor WRITES its own field (v_ = 99), NO operators    — the ORDER OF CONSTRUCTION (W).
        A printing ctor shows a wrong ORDER; only a WRITING one shows a wrong ANSWER. It has
        no operators on purpose: the DEFAULT copy was the broken path.
  Zord  the same writing ctor, PLUS op= and a 2-arg op+      — the ORDER IN A SLOT (Z7). Ord
        cannot be a chain operand (no operators), so the slot-wise explode needed its own
        observer: a chain landing in a tuple slot must be CONSTRUCTED before it is written
        into, or it reads back 99 instead of the sum.
  Val   silent, KEEPS its ctor argument                      — the SOURCE MODEL (Y). Y asks
        which source got read and how many times, so it reads VALUES back; Ord can't be used
        (its ctor deliberately overwrites what was passed).
Blocks: J (the baseline, 12 -> 7), K (the ladder), L (the trivial bucket), M (destinations +
shapes), N (the un-fusable buffer + its lifetime), O (the three transfer gates), P (destructure
from a literal: one accumulator per slot, moved in), Q (where the operator comes from: inherited
/ virtual / method body / ctor+dtor body), R (the other kinds: shift, logical; + a global target,
a cross-class destination, switch and do-while bodies), S (for-long + for-ranged bodies, a heap
target and a heap-deref operand, destructure into declaring slots), T (the blast radius of the
derived->base bind: exact beats the upcast, a base-only overload is reachable, virtual dispatch
survives, a pointer-field move), U (the arity-1 unary as a chain node), V (a tuple of classes:
built in place from a literal of constructions, op= per element from a literal of lvalues, and
nested destructure in all three forms), W (the ORDER of construction — every binding copied from
an lvalue, read back as a VALUE), X (every spill dies at the SEMICOLON — each case prints the
temp's dtor BEFORE its marker line), Y (the SOURCE MODEL — a nested rvalue source, a counted
source evaluated exactly once, a spilling destructure per loop iteration, a discard slot whose
element still runs, an array copy-init from an array lvalue, a copy-init in a NESTED function),
Z (an operation on an AGGREGATE of classes -- tuple+tuple, array+array, the aug-assign's
per-slot `op+=`, a scalar class BROADCAST, nested, and a spilled call operand; Acc prints, so
the objects are countable: each slot costs one accumulator, moved in and dead at the semicolon
-- and Z2, the ARRAY form, now costs exactly what Z1's TUPLE form does: it is the regression
test for the explode retyping an array as a tuple, which made its binding a CROSS-FORM copy
and spilled the whole aggregate to a temp, a ctor and a dtor per slot)
and Z7 (the ORDER of construction IN A SLOT, class Zord -- a wrong ANSWER, 99 not the sum),
Z8/Z8b (a construction head whose COLLAPSE IS DECLINED -- class Buf has the 2-arg op+ and no
op+=, so it cannot collapse; it costs ONE temp, dead at the semicolon, at the top level AND
inside a slot. K3 pins the collapse being TAKEN: Str has op+= and no 2-arg op+, so it MUST)
and Z9 (a CONSTRUCTION as the BROADCAST operand: read by every slot, not re-readable, so it
is SPILLED and built exactly once -- never cloned, which would strip its is_construction).
Negative: neg_aug_no_op. Full detail in readme-classes.txt "CLASS-OPERATOR CHAINS".

THE ORDER OF CONSTRUCTION — FIXED 2026-07-13 (it was filed here as a deferred wart, and was
                                              in fact a WRONG ANSWER)
  A COPY-INIT ran op= and THEN the ctor: the funnel FILLS storage and then FINALIZES it, so a
  fresh decl from an lvalue had its CONSTRUCTOR RUN ON TOP OF THE COPY. It was filed as
  cosmetic ("uniform, value-correct, nothing observes the gap") because every counting class in
  evaluate.sl only PRINTS in its ctor — a printing ctor shows a wrong ORDER; only a ctor that
  WRITES its own field shows a wrong ANSWER, and there wasn't one anywhere in the suite. There
  is now: class Ord, block W. `D dx = dy;` yielded 99 (the ctor's value), not dy's.
  THE RULE: a class can only be COPIED INTO, so it must EXIST first — alloc, init, ctor, THEN
  the transfer. classify peels the transfer off the declaration (applyTransferSplit) and
  re-emits it against the constructed object, per SLOT. The user's framing, which is the whole
  design: "an object is copied into; a thing that does not exist yet cannot be copied into."
  WHAT THIS IS NOT: it does not touch the ELIDE. What decides whether a copy happens at all is
  the SOURCE — an rvalue has no object to copy FROM, so it builds in place, and `(Ord(1), wa)`
  still builds slot 0 and copies only into slot 1.
  THE PART THAT DOES NOT GENERALIZE: a class FIELD from an lvalue. A construction's arguments
  are FIELD INITIALIZERS, and a ctor must see its fields ALREADY INITIALIZED — Q4's Hook
  computes `a_ + b_` in its ctor body and must read what was PASSED, not the defaults. Hoisting
  a field's copy past the enclosing ctor breaks that (Q4 caught it, printing 0 instead of 3).
  The copy has to land BETWEEN the field's own ctor and the enclosing ctor body — inside
  emitConstructed's hook recursion, where the initializer expressions no longer exist. Until
  that interleave lands, the case is REJECTED, not mis-ordered (2026-07-15, "cannot be
  initialized by copying/moving", + the value-init "cannot dispatch Class.op="); the copy/move
  gate uses userSelfTransferOpId so a trivial-bucket field class (op-only, no ctor/dtor) is
  caught too. See readme-classes.txt "A CLASS FIELD THAT WOULD NEED A TRANSFER".
  THE OTHER PLACE IT DID NOT REACH: a GLOBAL (`global Add gb = ga;` read back 205, not 105).
  Not because the rule differs, but because a global declaration is not lowered where locals
  are: its initializer becomes the body of a synthesized lazy ctor in desugar, and there is no
  statement list at the declaration to split the transfer into. Found while writing block Y's
  tests. CLOSED 2026-07-13 by REJECTION, not repair: a global's initializer must be a CONSTANT
  EXPRESSION — data, not code — so a global class can only be built from a constant field list,
  which builds IN PLACE. There is no copy left to order wrongly. The lesson is worth keeping:
  the fix for a bug is not always in the machinery that produces the wrong answer. Ask first
  whether the shape that reaches it should exist. (plan.txt, A GLOBAL'S INITIALIZER IS DATA.)
  Block R3's `gacc` moved to the declarator form (`global Acc gacc(100);`) — `= Acc(100)` is a
  construction EXPRESSION and is no longer a legal global initializer. Same object, same golden.
  THE THIRD PLACE IT DID NOT REACH: the RETURN SLOT — FIXED 2026-07-13, and it belongs to
  return_fn.sl, not here. Every NON-NRVO sret return (a tuple of class locals, overlapping
  locals, `return p^`) moved into the caller's slot and then ran the ctor on top. I had filed it
  as an open design question; it was not one — return_fn.sl's canon case 3 already dictates
  `initialize ret^; ret^.ctor(); ret^ <-- a;`. A returned LITERAL is bound to a `_$ret` local so
  the declarator funnel has a target (NRVO then builds it in the caller's slot — free); a
  whole-value LVALUE return is ordered in codegen from the DEFAULTS classify parks on the return
  node. NRVO is the ELIDE of that pair and is untouched. Z7 pins the DECL half of the same rule;
  the RETURN half is pinned by return_fn.sl's class Ret. (readme.txt / readme-classes.txt.)

DESTRUCTURE'S COST — CLOSED 2026-07-13 (was the open item here)
  Block P used to build 4 objects where 2 would do, because a non-bare-name source was spilled
  to a `_$dsrc` tuple temp and each chain was then copied TWICE (into the spill slot, then into
  the target). The fix was not in the chain machinery at all — it was the destructure's SOURCE
  MODEL (readme.txt, DESTRUCTURE — THE SOURCE MODEL): a tuple LITERAL is now taken apart and
  each element binds straight to its slot, so each slot is a chain with a destination. A LIVE
  slot gets one statement-scoped accumulator MOVED in (op<-- finally runs here); a DECLARING
  slot IS the accumulator and costs nothing at all (block S4). The spill survives only where it
  earns its keep — an rvalue source, or a literal that reads a target (`(sa, sb) = (sb, sa)`
  must swap). The LIFETIME half had already been fixed by [[project_stmt_scoped_rhs_temps]].
  (NOT a gap: TEMPLATES. v2 has NO template machinery at all — plan.txt Phase 9, unstarted.
   CLAUDE.md used to describe v1's template system as if it existed here, which is why earlier
   drafts of this plan listed ".slh / template context" as an untested surface. Corrected
   2026-07-12; the v1 design was migrated to v1/v1.md → "Templates". There is nothing to test.)

TWO BUGS THE COVERAGE SWEEP FOUND (both PRE-EXISTING, neither caused by the chain work)
  - AUG-ASSIGN HAD NO NO-VIABLE-OPERATOR GUARD. `a += b` on a class with no `op+=` fell
    through to the NUMERIC path, where commonType SUCCEEDS for two identical class types —
    emitting a struct `add`. INVALID IR, from a class with no operators at all. The
    kBinaryExpr arm has had this guard for a long time; kAugAssignStmt never got it. Fixed;
    negative neg_aug_no_op in evaluate.sl.
  - AN INHERITED OPERATOR WAS UNREACHABLE. Its params are typed as the BASE, so a derived
    operand must bind a `Derived` VALUE to a `BaseOp^` param. argConvertCost's auto-ref arm
    accepted only an exact class or a per-leaf widen and returned -1 — so every user-written
    base operator was dead from a derived operand (and `Derived += Derived` then fell into the
    bug above and emitted invalid IR). It is an UPCAST, not a conversion: the base is the
    derived's slot-0 sub-object, the address is identical, nothing is sliced. Fixed in three
    places — argConvertCost (rung 2, matching what the POINTER arm already gave `^derived`),
    a new shared checkArgAssign (the three arg/param check sites now agree), and emitCall
    (a class lvalue passes BY ADDRESS, never through a materialized temp of the BASE's
    layout — which is what produced the second round of invalid IR). Canon evaluate.sl Q1.
