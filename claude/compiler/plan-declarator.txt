compiler — UNIFIED DECLARATOR — ALL STAGES (design + STATUS)

A declarator binds a NAME to a TYPE with an optional INITIALIZER — where the
initializer is an assignment-like operation: copy `=`, move `<--`, swap `<-->`,
or compound `OP=`. The design goal is that this ONE concept funnels through ONE
path PER STAGE (parse -> resolve -> classify -> desugar -> codegen), instead of
being re-implemented per binding site / per statement-kind.

DONE — all five stages now funnel: parse (parseDeclarator), resolve (resolveDeclType
+ registerDeclarator), classify (dispatchAssignInit + desugarDestructure), desugar
(lowerAggCopyStmt), codegen (emitInitFill / emitConstructed / emitConstructAt). The
scatter that made behavior drift between sites and made "we missed a cell" bugs recur
(the kStoreStmt op= hole was the canonical example) is gone — each stage has ONE path.
The rest of this file is the design of record + the per-stage landing notes.

======================================================================
THE PRINCIPLE, ALL STAGES
======================================================================

A binding SITE is one of: local var-decl, function parameter, for-loop variable,
destructure slot, tuple-type slot, return value, and the assignment-form targets
(bare-name `x = e`, complex-lvalue `obj.f = e` / `p^ = e` / `arr[i] = e`), plus
the move / swap / compound forms of each. At every stage these are DIFFERENT
parse-node kinds, but they are the SAME declarator concept and must share:

UNNAMED VARIABLES AND TEMPS COUNT. The codegen half of the umbrella
(emitInitFill / emitConstructed / emitConstructAt) is really "MATERIALIZE A VALUE
INTO STORAGE (+ ctor hooks, + optional dtor registration)" — it is NOT limited to
NAMED declarators. Any site that constructs / fills a value into storage belongs
to it, whether or not a name is bound: a heap object (`new T(init)`), a conversion
temp (`_$cret`), an operator-result temp (`_$optmp`), a materialized call-arg
rvalue. Routing these through the ONE funnel is what keeps the field-init / ctor /
dtor triad from drifting per site — the same "we missed a cell" guarantee, extended
to the nameless construct sites.

TWO FEED PATHS reach the three helpers, and EVERY nameless site was once hand-rolled:
  (1) DIRECT — a codegen construct site calls them itself: var-decl / sret / global /
      `new` / the call-arg pass temp.
  (2) VIA THE LIFT — classify+desugar rewrite a nameless class temp (`_$cret`
      conversion, an inline `Class(args)`) into a NAMED
      kVarDeclStmt (liftSretCallExprs, desugar.cpp), which then takes path (1)'s
      var-decl route.
      NOT the operator-result `_$optmp` — not any more. classify used to lower a class
      operator into a `_$optmp` + a class_conversion lift (lowerClassOperatorTemp), and
      that reached the funnel through THIS path. Since the chain lowering (2026-07-12,
      and the arity-1 UNARY with it 2026-07-13 — lowerClassOperatorTemp is deleted),
      DESUGAR mints the accumulator itself as a named kVarDeclStmt (lowerOpChain /
      makeAccDecl), so it takes the var-decl route directly — and often there is no temp
      at all, because the DESTINATION is the accumulator. See readme-classes.txt
      "CLASS-OPERATOR CHAINS". Verified: a `(Money = 5)` conversion emits @Money__$ctor + the
      op= fill + a REGISTERED dtor at scope end — exactly like a named local. A
      lifted temp that is a NESTED arg / receiver in a var-decl / assign / return rhs
      whose VALUE is a SCALAR instead folds into a kSeqExpr wrapping the rhs (statement
      lifetime, register_dtor=false — codegen's kSeqExpr owns teardown), 2026-07-11.
The path-2 class temps were funneled by an EARLIER session (the lift IS their fold);
`new` (its own kNewExpr codegen) and the call-arg temp (inline in emitCall) were the
two hand-rolled DIRECT sites THIS work folded. The umbrella is now COMPLETE — every
construct/fill site, named or nameless, direct or lifted, goes through the funnel.

SITE x OPERATOR MATRIX  (classify: does the site dispatch the USER operator?)

  site (statement-kind)          op=      op<--    op<-->   compound
  ---------------------------    ----     -----    ------   --------
  decl-init   (kVarDeclStmt)      yes*     yes*     yes*     n/a
  local =     (kAssignStmt)       yes      n/a      n/a      n/a
  complex =   (kStoreStmt)        yes      n/a      n/a      n/a   (Slice A: was HOLE)
  move        (kMoveStmt)         n/a      yes      n/a      n/a
  swap        (kSwapStmt)         n/a      n/a      yes      n/a
  compound    (kAugAssignStmt)    n/a      n/a      n/a      yes
  destructure copy                yes      n/a      n/a      n/a   (per-slot kAssignStmt)
  destructure move/swap           n/a      yes      yes      n/a

  EVERY binding site now routes through the ONE funnel dispatchAssignInit (UNIFIED
  2026-07-05): the 5 assignment-form sites AND both decl-init shapes — typed `Class x=e`
  and INFERRED `x=e` (`cls = create()`). The guard `form==kSlid && findClassOperator>=0
  -> rewrite` exists in ONE place, so a new site cannot skip it. Two knobs let a DECL
  target share it: `prelude` + `needs_construct`. On a match the funnel SPILLS a class
  RVALUE source (call/op result) into a `_$cinit` temp built by the default elision so
  the operator's `Class^` param gets an address (a construction source is spilled only
  for a fresh decl; a live-assign construction bails to the clean "construct in this
  position" error), and — if `needs_construct` — splices a default-construct of the
  target before the rewrite. ELIDE-WHENEVER-POSSIBLE (2026-07-08, return_fn.sl site canon):
  a DECL-INIT (`needs_construct`) from a same-type class RVALUE now ELIDES — the funnel
  bails (`isBareLvalue==false && exact`) so the call/construction builds in place, no op.
  The operator fires only where elision can't apply: a LVALUE copy (`Copy ca = cb`), a
  NON-EXACT convert (`Dst d = mkSrc()` -> op=(Src^)), or an EXISTING-var assign (case 3/4).
  This REVERSES the 2026-07-05 "a user op wins over elision even from an rvalue" seam (the
  author forces the op by declaring, then assigning). Inferred `x=e` follows the same rule.
  Side effect still holds: the assign-from-call-rvalue segfault (`x = mkCopy()`) stays closed.

  DECLARE-IF-FRESH now covers ALL FOUR init shapes (2026-07-05). Copy `x=e` (kAssignStmt)
  already promoted a bare undeclared target to a fresh inferred decl; MOVE `x <-- e` and
  SWAP `x <--> e` did NOT — they resolved the target as an existing lvalue and errored
  "Cannot assign to undeclared variable 'x'". Now resolve's `tryInferredMoveSwapInit`
  (before resolveStmt) mirrors copy: a bare UNDECLARED, NON-field move/swap target declares
  a fresh inferred local and flips the stmt to a kVarDeclStmt with default_move_init /
  default_swap_init (a field or an existing var falls through to the normal move/swap).
  classify already dispatches op<-- for the inferred move; the swap-init block now infers
  its target type from the source before the class-type check. So `cls <-- create()`,
  `x <-- y`, `nw <--> ex` all init-from-fresh, symmetric to `x = e`.

PER-STAGE CLEANUP TARGETS  (audit evidence — file:line as of this audit)

RULING (for-vars + all declarations, from the user — REVISED): every VARIABLE DECLARATION
behaves the same way everywhere, incl. shadow checks — one funnel, no per-site surprises.
All four for shapes obey NORMAL LEXICAL SCOPING (short-for scopes: enclosing > loop-var >
body; long-for adds a sibling update scope), so a TYPED decl may shadow an enclosing
same-name entry — local OR non-local (const / fn / class) — legally; only a same-frame
collision (two names in one varlist) is a "Duplicate declaration". A TYPELESS binding needs
a type source (an initializer / iterable): if the name already resolves it is a REUSE — an
assignable variable is reused, a non-assignable one (const / class / enum / fn) is REJECTED
"Cannot assign to <noun> '<x>'." exactly as an assignment would be (so `for (c : t)` with a
const `c` == `c = ...`; canon fortuple.sl neg_var_shadows_const, tuple/destructure.sl
neg_dup_const); unresolved -> a fresh inferred local. A TYPELESS binding with NO type source
(a bare `for (x)`) is a structural error "A variable declaration needs an explicit type or
an initializer." regardless of what the name resolves to (`int x; for (x)` still errors — a
varlist slot may not be a bare "touch"). Canon: flow/for{long,array,tuple,range}.sl (the
`for (x)` reuse-without-init form + its neg_typeless_const/function tests were REMOVED as
wrong interpretation; the top-of-file comment is canon). LANDED: bindForVar (array + tuple)
+ the long-for varlist + inferred-init all route the typeless decision through registerDeclarator;
the AnyScope conflict-mode + the existing_local reuse gate + the hand-rolled long-for-no-init
branch are all gone.

RULING (Slice C rationale, from the user): ctor/dtor are ALWAYS balanced; the way to make
unbalance impossible is to route every construct/destruct through the SAME path — so
emitConstructAt OWNS dtor registration and no emit site can forget it.

MOTIVATION

Today a "type position" is parsed by two cooperating routines: parseType for the
type expression, parseNameDims for the array dims that hang off a declared name.
The reject_array_dims bool is really a proxy for "is there a name here" — at a
named decl the sized dims go on the name (int x[3]); at a nameless site they go
inline (return type (int,int)[3]). That split is why a nameless tuple SLOT can't
spell a composed type (int^[2]) — there is no name to hang the array on, and the
inline suffix parser only accepts ONE modifier category.

There are 10 type-parse sites (grammar.cpp) plus destructure. Unify them.


THE DECLARATOR

One production, parameterized by a per-site name policy:

    declarator(name_policy) := [qualifier] [core-type] [name] [name-dims]

  qualifier  : a prefix `const` / `mutable` (Phase 6; see QUALIFIERS).
  core-type  : OPTIONAL — a base + a left-to-right chain of postfix wrappers.
               Omitted = a TYPELESS declarator: the type is INFERRED from the
               binding. Four binding contexts: a var-decl `= init` (`x = 5`), a
               param `= default` (`f(x = 5)`), a for-var `: iterable`, and a
               destructure slot that DECLARES. A nameless TYPE site (return type,
               alias RHS, ...) requires the core-type. (A typeless binding to an
               already-resolved name — a plain `x = 5` or a destructure slot that
               REUSES an existing variable — is an assignment, not a declarator;
               the declare-vs-reuse split is by lexical resolution.)
  name       : an identifier; presence governed by name_policy.
  name-dims  : the OUTERMOST sized-array dimensions, written on the name.

At least one of {core-type, name} is present. BOTH omitted is the EMPTY
declarator — a destructure DISCARD slot (only valid in a ListSlot). The
`ident ident` vs `ident =`/`ident;` lookahead splits "core-type + name" from a
typeless "name" (a lone trailing identifier is the name, not the type).


CORE-TYPE — THE WRAPPER CHAIN

    core-type := base { ^ | [] | [N] }*
    base      := primitive | ident(alias/class/enum, possibly qualified)
               | ( declarator-list )            # a tuple type

Each postfix wrapper applies to the type ACCUMULATED SO FAR, left to right:

  ^      reference   (no arithmetic)
  []     iterator    (arithmetic allowed)
  [N]    sized array dimension

So the order of suffixes is the order of nesting, and every composition is
spellable without parens:

  int^[3]   ==  (int^)[3]   == array of 3 references
  int[3]^   ==  (int[3])^   == reference to an array of 3
  int[2][3] ==  (int[2])[3] == row-major: 2 of (3 of int)

The current single-modifier suffix (the if/else-if/else-if) becomes a LOOP over
wrappers. `[]` (empty brackets) is the iterator wrapper; `[N]` (with a size) is a
dimension — the parser already distinguishes them by emptiness. A `^^` (lexed as
ONE logical-xor token) counts as TWO reference levels in the loop, so `int^^`
spells a pointer-to-pointer; a 3+ caret run already arrives as single `^` tokens.

No parens are EVER required to disambiguate the postfix chain (postfix,
left-associative, no precedence to resolve). Parens are only needed to scope a
prefix qualifier (see QUALIFIERS).


THE NAME, AND name-dims — "TOP-LEVEL SIZED DIM ON THE NAME"

The name is optional FOR THE PARSE; whether it is required/forbidden is the
per-site name_policy. Where a name is present, the rule for sized dims is:

  At a named declarator, the OUTERMOST sized-array dimension is written as
  name-dims, not inline. Sized dims NESTED inside a pointer wrapper (or hidden
  behind an alias) stay in the core-type.

Worked across the cases (declaring v):

  int v[3]      core int          name-dims [3]   OK    v : int[3]
  int[3] v      core int[3]  (top-level dim)       ERR   -> int v[3]
  int[3]^ v     core (int[3])^ (dim inside ^)      OK    v : ref-to-int[3]
  int^[3] v     core (int^)[3] (top-level dim)     ERR   -> int^ v[3]
  Vec3 v        core Vec3 (dim hidden in alias)    OK    v : int[3]
  Vec3^ v       core Vec3^ (outer is ^)            OK    v : ref-to-int[3]

The "size belongs on the name" discipline survives in this sharper form: only a
TOP-LEVEL inline sized dim is rejected. An alias is one identifier token carrying
no textual top-level dim, so `Vec3 x` parses and resolve composes int[3]; the
naive substitution `int[3] x` being illegal is irrelevant (substitution is on the
resolved type, never on tokens). name-dims compose on top of an alias's dims:
`Vec3 x[2]` -> int[2][3] (2 of int[3]).

DECL-RECOGNITION GATES (upstream of the above). Before parseType runs, a `(`-led
or identifier-led statement must be RECOGNIZED as a typed declaration vs a
statement. The two lookaheads that do this — looksLikeTupleTypeDecl (`(`-led) and
looksLikeQualifiedTypedDecl (identifier/alias-led) — share ONE tail,
typeSuffixesThenName: from just past the core type it skips a MAXIMAL interleaved
run of type-suffix tokens (`^`/`^^` and bracket groups, empty `[]` OR sized `[N]`,
by bracket depth — mirroring parseType's real chain), then requires the trailing
name. (A primitive-led decl needs no lookahead — isTypeStart commits it straight
to parseType, which is why primitive-led sized types never had this gap.) The
trailing-name check is the sole decl-vs-statement discriminator: a store/call puts
an OPERATOR after the suffix run, so `arr[i] = v` / `p^ = v` stay statements
regardless of bracket emptiness. Consequence: a `(`-led or alias-led type with a
NESTED sized dim (`(const int)[5]^ p`, `Vec[5]^ v`, `(int,int)[2]^ t`) now parses
as a decl (previously the gate stopped at the sized `[` and misparsed it), and a
TOP-LEVEL sized dim (`(const int)[5] x`) is recognized as a decl and reaches the
"size belongs on the name" diagnostic above rather than a confusing statement
misparse. Tests: array.sl (`grp`/`ali` var decls + neg_array_type_grouped/alias),
call.sl (sum_grp_ref/sum_alias_ref/sum_tup_ref params + neg_array_param_grouped/
alias).


name_policy PER SITE  (replaces reject_array_dims)

  Required   var-decl declared type, parameter, for-loop variable
  Forbidden  return type, alias RHS, enum underlying, cast target,
             conversion target, sizeof operand
  ListSlot   destructure / tuple-type slots — name PARSED as optional; the
             POSITION decides. A tuple TYPE forces its slots ANONYMOUS (a named slot
             is "too many names") and a destructure NAMES them — the by-position
             rule, NOT a Forbidden parse policy (see TUPLES AND DESTRUCTURE).

A parameter name is REQUIRED — there are no anonymous parameters.


TUPLES AND DESTRUCTURE — A PARENTHESIZED `( ... )`, READ BY POSITION

`( ... )` is the shared shape for a tuple TYPE and a destructure, but the rule is
decided by POSITION — not by one unified name-counting scheme:

  TYPE position (var-decl / param / return): the slots are TYPES and must be
  ANONYMOUS. A NAMED slot is "TOO MANY NAMES" — the name belongs on the variable,
  not inside the type:
       (int, int) z          OK     z : (int, int)
       (int x, int y) z       ERROR  "too many names" (x,y named in the TYPE, plus z)

  DESTRUCTURE (`( ... ) = expr`, dispatched by the trailing `=`): the slots are
  BINDINGS and carry as many names as needed, at ANY nesting depth — "too many
  names" never applies:
       (int x, int y) = e            OK
       ((int a, b), int c) = e        OK   (nested; names at every level are fine)

So "too many names" is purely a TYPE-position rule (names where only types belong);
a destructure is the `=`-bound form, and it nests freely:

  ((a, b), c) = ((1, 2), 3)      # (a,b)<-(1,2), c<-3  =>  a=1 b=2 c=3


DESTRUCTURE SLOTS

A destructure binds a `( slot, ... ) = expr`. Each slot, dispatched on
type-presence, obeys the ordinary scope rules — a slot is exactly a var-decl OR
an assignment:

  typed   (int x)   declare in the CURRENT scope; same-scope collision is a
                    dup-decl ERROR; an enclosing same-name is shadowed (legal).
  untyped (x)       lexical-resolve the name: a reusable LOCAL anywhere up the
                    chain -> ASSIGN (reuse); a same-frame NON-LOCAL (const / fn /
                    class) -> dup-decl ERROR (not a silent shadow); found nowhere ->
                    declare-inferred in the current scope.
  nested  ((a,b))   a `( sub-list )` slot -> recurse, matched against the
                    corresponding nested tuple value (per-slot rules apply at the
                    deeper level; names may appear at any depth).
  discard           an empty slot (a bare `,`) OR a TYPED slot with NO name (`int`)
                    -> drop that tuple position. The slot name is OPTIONAL; a
                    typed-no-name slot is a documented drop, same as empty.

ONE NAME PER DESTRUCTURE: a name may target at most once across the WHOLE pattern
(including nested sub-patterns) — a repeat is a duplicate-target ERROR whether it
would declare (`(y, y)`) or reuse an existing variable (`(a, a)`). Tracked by a
per-statement seen-set (resolve), separate from the same-frame dup-decl check.

Arity must match the tuple width INCLUDING discards; the trailing comma is
SIGNIFICANT (it adds a discard slot). Worked example:

    int a; int b;
    { int c;
      (a, int b, int c, d, ) = foo();   // foo returns a 5-tuple
    }
  a      untyped, resolves (outer)         -> reuse  (assign into outer a)
  int b  typed, b only in outer scope      -> new var in the block (shadows)
  int c  typed, c in THIS scope            -> ERROR  (same-scope dup)
  d      untyped, unresolved               -> declare-inferred
  (last) empty                             -> discard

Targets are bare names (declarator-shaped), NOT arbitrary lvalues — `(arr[i], b)
= e` is out (that was the "tuple list of lvalues" we rejected).


QUALIFIERS  (const / mutable — parse + munge + casts LANDED; enforcement Phase 6)

A prefix qualifier binds LOOSEST: it wraps the ENTIRE suffix chain to its right.
Parens scope it to an inner sub-type — the one place parens ARE required.

  const int^      ==  const (int^)   == const reference to int
  (const int)^    ==  reference to const int

  precedence:  postfix suffixes bind tight (left-to-right);
               a prefix qualifier binds loose (wraps everything after it).

const is a tracked TypeRef qualifier (Form::kConst) — the placement above IS the
deep/shallow distinction. It is TRANSPARENT for matching (strip/deepStrip see
through it) but VISIBLE in spell(); ENFORCEMENT defers with the const-correctness
work. Round-trip: intern() peels a leading `const ` FIRST (loosest), and the
grouping parens ride the size-1-tuple collapse ((T) -> T) — so `(const int)` stays
parenthesized only when const-bearing, keeping a following suffix OUTSIDE it
(shallow). `mutable` is a pointer/iterator PARAMETER qualifier only (rejected
elsewhere). The only first-class use of `(...)` in a type stays the tuple
constructor (with commas). Canonical spellings are paren-free where the chain
suffices (int[3]^, not (int[3])^).


OPERAND-SIDE ALGEBRA  (validates the type design; not new code)

The dereference / subscript operators are the inverse of the wrappers:

  ^      (postfix deref) peels a POINTER wrapper (^ / []); requires pointer-like.
  [i]    (subscript)     peels an array dimension.
  ^      (prefix addr)   adds a wrapper: addr-of a bare lvalue -> reference (T^);
                         addr-of an indexed element -> iterator (T[]).
  ^^     (either side)   two carets: `p^^` peels two wrappers; `^^x` adds two
                         (nonsensical — same operand-error as `^^^x`).

  int[3]^ v;   v       : int[3]^   the reference (base address of the array)
               v^      : int[3]    peels ^  -> the array, located at that address
               v^[0]   : int       peels a dim -> element 0
               ^v^[0]  : int[]     addr-of an indexed element -> iterator
               v, v^, ^v^[0] share one numeric address (&array == &array[0]).

  int x[3];    x^      ILLEGAL — x is an array, not a pointer; deref needs a
                       pointer wrapper. The first element is x[0].

Array -> pointer DECAY is now implicit (LANDED 2026-07-09, [[project_array_pointer_decay]]):
a bare array lvalue in a POINTER target decays to its ELEMENT pointer `^arr[0]`
(`int[] q = arr`, `int^ p = arr`, iterator/element-ref args) — classify rewrites it
to the address-of. The WHOLE-array ref `int[N]^` is NOT implicit from a bare array
(write `^arr`); `x^` on a bare array stays ILLEGAL (deref needs a pointer wrapper).
`^` <-> pointer wrappers, `[i]` <-> array dims, still kept distinct in the grammar.


PASSING — PRIMITIVES BY VALUE, EVERYTHING ELSE BY POINTER  (invariant since day 0)

A PRIMITIVE parameter passes by value (copied). EVERYTHING else passes BY POINTER —
the pointers (reference `^`, iterator `[]`) AND arrays/tuples/classes. There is no
by-value/copy form for a non-primitive, and a non-primitive VALUE parameter is a
COMPILE ERROR:

  void f(int a)         primitive             -> by value
  void f(int[] p)       iterator (a pointer)  -> by pointer
  void f((int,int)^ t)  reference (a pointer) -> by pointer
  void f(int a[3])      ARRAY shorthand       -> by pointer  (see below)
  void f((int,int) t)   tuple BY VALUE        -> COMPILE ERROR
  void f(Vec3 v)        array BY VALUE        -> COMPILE ERROR  (Vec3 = int[3])

THE ARRAY SHORTHAND. `int a[3]` and `int[] p` as PARAMETERS are both pointers —
`int a[3]` a pointer to a KNOWN-size array, `int[] p` one of unknown size. (As a
VARIABLE, `int a[3]` is three ints — array-as-data; as a PARAMETER it is the
address — array-as-pointer. Same tokens; position disambiguates.) `int a[3]` is
sugar for `int[3]^ a` but indexes WITHOUT a deref (`a[0]`, not `a^[0]`); the
by-pointer-ness is invisible to the author.

DECLARATION vs CALL — value-spelling is a CALL-SITE convenience, NOT a param form.
The declaration is always the pointer (`void f(String^ s)`). The CALL is either:
  f(^s)   explicit address-of.
  f(s)    the convenience — semantically identical to `f(^s)`. Two lowerings:
          LVALUE s (a named local / field / index / deref) — pass &s directly,
            no copy. The function mutates the caller's data.
          RVALUE s (a call return, op+ result, tuple literal) — materialize the
            value in a stacksaved alloca and pass that alloca's address (so
            f(String + "x = " + x) still has a stable address to take).
There is no value SPELLING of a non-primitive parameter; the "value" look lives
only at the call site (and in the array shorthand, which still passes a pointer).

mungeParamTypes — the PARAM-TYPE REWRITE HOOK (resolve), over each resolved param:
  primitive          -> pass through (by value)
  array  (kArray)    -> by pointer (to a CONST array unless `mutable`): semantically
                        deref'd at use, INVISIBLE to the author (codegen receives a
                        `ptr` and binds the param's alloca to it — no copy, no `^`)
  pointer (^ / [])   -> pointer-to-CONST pointee unless `mutable`: `int^ p` becomes
                        `(const int)^ p`, `int[] q` becomes `(const int)[] q`
  everything else    -> COMPILE ERROR (a non-primitive parameter must be a pointer
                        or an array)
All arms are LIVE: the pointer/array pointee-const rewrite runs (const is now a
real TypeRef qualifier), `mutable` opts out, and the synthesized method receiver
`_$recv` is SKIPPED (const methods are Phase 6). `##type` shows the munged type.
Still INERT until Phase 6: ENFORCEMENT (nothing rejects a write through the const
pointee yet) — see plan.txt Phase 6.

Call side: emitCall splits the by-pointer arg into the lvalue arm (pass &arg
directly via emitLvalueAddr) and the rvalue arm (an alloca FILLED via emitInitFill —
the ONE fill funnel — freed by the stacksave/stackrestore bracket, pass the temp's
address). autoRefPointee (classify) supplies the
TYPE side of the same convenience: a non-primitive value arg checks against the
param's pointee.

     also makes sizeof(tuple) report a real size + agrees with codegen's malloc size.

+ THE CONVERSION-TARGET TEMP (LANDED): the 12th site. A value conversion `(Type = src)` IS
  a declarator — an ANONYMOUS one, in expression position — binding a fresh temp of the
  target type from `src` (the header's own framing: "type conversion is assignment to a
  temporary variable"). For a PRIMITIVE (scalar) target the temp is elided to an SSA
  value (widen::convertExplicit). An AGGREGATE target desugars BY SLOT in classify
  (lowerAggregateConversion): the kConvertExpr BECOMES a tuple of per-slot sub-conversions
  `((T0=src[0]), (T1=src[1]), ...)`, each re-inferred — a class leaf at any depth reuses the
  class path below, a primitive slot the scalar path, a nested aggregate recurses; cross-form
  (array<->tuple) and a spilled side-effecting source (a `_$cinit` temp evaluated once,
  agg_conv_spill, hoisted by desugar) fall out. For a CLASS target the declarator is real:
  classify's kConvertExpr arm calls lowerClassConversion, which — reusing the SAME machinery a
  decl-init op= uses — default-constructs a `_$cret` (classifyClassInit) and FILLS it from the
  source: it dispatches `_$cret.op=(src)` (findClassOperator + makeOpCallStmt). findClass-
  Operator surfaces every op= — a user op=(T)/op=(T^) AND the synthesized default op=(Self^)
  — so a same-class source resolves to the memberwise default copy with no bespoke fallback
  (the old same-class kAssignStmt hack is DELETED). Stashes [construct, fill] as the node's
  two children (class_conversion flag). Desugar's liftSretCallExprs lifts the pair into a temp
  (the class analog of a construction lift) and yields a `_$cret` read; the temp constructs /
  destroys through the existing class-temp lifetime (phrase-scoped at a statement-expression
  position, enclosing-scoped at a decl/assign rhs). ONE small codegen touch: in a CONDITION
  phrase the pair lifts through the kSeqExpr path (lowerPhraseSlot, not the statement-pre
  path), so the seq-child emit loop grew a branch to DISCARD a void effect call (the op=) via
  emitCall — otherwise emitExpr asserts on a void value. The grammar reaches a class target via
  looksLikeConvTarget (the factored lookahead): a top-level `=` inside `(...)` is unambiguous
  (slids has no assignment-EXPRESSION); it scans `::` (global) + `:` (member) segments, so a
  NAMESPACED target (`Money:Cents = x`) routes too. A source that is neither op=-viable nor the
  same class is a clean reject. Test: test/assign/typeconv.sl — scalar value / assign / pointer
  / class-source / in-place / return / multi / namespaced / alias / virtual / condition, plus
  the aggregate surface (class-leaf tuple, same-class copy, mixed, class array, cross-form
  reshape, nested, spill, cross-class-in-slot, virtual-in-slot, chained, class-typed-field copy)
  + negatives (no-op= / pointer-target / abstract). REMAINING: a class conversion inside a
  NESTED FUNCTION crashes — a pre-existing desugar-pass gap, not conversion-specific (todo.txt).

SETTLED

- parameter name: REQUIRED (no anonymous parameters).
- params: PRIMITIVE by value; everything else BY POINTER. A non-primitive VALUE
  param is a COMPILE ERROR — the declaration is always the pointer (`String^ s`),
  and value-spelling (`f(s)`) is a CALL-SITE convenience. Sole exception: the array
  shorthand `int a[3]` (a pointer, indexed without a deref). Done via the
  mungeParamTypes hook (resolve); the pointer-to-const arm is LIVE (`mutable`
  opts out), only its ENFORCEMENT defers to Phase 6.
- untyped destructure slot resolving in an enclosing scope: assign into the outer
  variable (reuse).
- BOTH pointer kinds are RESEATABLE and NULLABLE (long-standing spec): `r = ^y`
  repoints, `r = nullptr` clears. `^` and `[]` differ ONLY in arithmetic (`^` has
  none, `[]` has it) — both reseat, both null. The const-reference semantics
  (`const int^` locks the reseat) build on this.
