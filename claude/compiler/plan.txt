compiler — PLAN (the main quest)

Feature implementation order. Items are open or LANDED; updated as we make
progress. Outstanding bugs / reach goals / deferred side-quests live in
todo.txt, not here. Detailed per-stage current-state notes (what each stage
does today, stage by stage) live in readme.txt.

Each phase adds primitives the next phase depends on. widen.cpp threads through
every conversion site as features land; treat each new expr/decl/assign site
as a widen call-out.

PIPELINE — current shape

    lex -> numeric -> grammar -> resolve -> constfold -> classify
        -> desugar -> optimize -> codegen

  resolve owns: parse::Entry vector population, frame push/pop, ident
  resolution (resolved_entry_id stamping), wrong-kind diagnostics
  (assign-to-function, call-a-variable), duplicate-decl + signature
  mismatch + arity-mismatch + multi-arg-print-intrinsic detection,
  type-spelling validation (widen::isKnownType for declared / return /
  parameter types), namespaces + enums (lowered to entries here), and the
  orphan-function check (declared-but-never-defined). Two passes per scope
  (collect decls then walk bodies) so forward references work.

  classify is type inference + (Phase 3) overload resolution. Stamps
  inferred_type / op_type, rejects no-common-type binaries, non-coercible
  operands for `! && || ^^` (and the `&&= ||= ^^=` aug-assigns), an if / while /
  for condition not coercible to bool, non-numeric
  shift sides, bitwise on float, and return correctness (non-void must end in a
  return — recursing into a trailing block and a both-arms-return if/else; void
  can't return a value). Reads entries via resolved_entry_id but never mutates
  the symbol table or pushes frames.

    (lambda-lift to top-level; capture host locals/params by reference;
    current value at call time; self-recursion ok; deep nesting + sibling
    calls unsupported [compile errors]; captured host var must be
    definitely-assigned at the call — deferred per-call snapshot check)
    SCOPE (2026-07-12): a nested function may be declared in ANY SCOPE of its host —
    a block, an if-arm, a loop body, a switch case — not just the body's top level.
    It lives in the frame it is WRITTEN in (visible there incl. before its own
    definition; gone when the scope closes) and captures that scope's locals; two
    same-named nested functions in disjoint if/else arms are NOT a duplicate. Sound
    because a capture is the host alloca's ADDRESS, all allocas live in the ONE
    function frame, and the function can only be CALLED inside the scope where its
    captures are live. resolve's signature pre-pass + deferred-body pass BOTH run
    per SCOPE (out of resolveStmtList, like the local-class pre-pass beside them) —
    they used to scan a body's DIRECT children only, so a block-nested function was
    silently unregistered (calling it: "Unknown function"; NOT calling it: its body
    never resolved -> classify ASSERTED). And a function may be nested in a FUNCTION,
    a METHOD or an OPERATOR (and a ctor/dtor) — all of them ARE functions. There is
    no "method nested in a method": a nested function inside a method is just a
    FUNCTION, one that INHERITS `self` IF IT NEEDS TO (it names a field — a bare `v_`
    rewrites to `self.v_` — or names `self`; either way `self` is an ordinary local of
    the method's frame and captures like any other). Canon test/function/nested_fn.sl.
    Desugar runs all four lowering passes on a nested body too (collectFunctionDefs,
    per function, with its OWN return type) — they used to loop over PROGRAM-scope
    functions only, so a nested body was NEVER lowered and a bare `i++` there asserted.
    See [[project_nested_fn_desugar]].

    BINARY SIGNATURE RULE (2026-07-11, canon operator.sl 170-182) — a binary operator's
    FIRST parameter must be a REFERENCE TO THE ENCLOSING CLASS: `op+(Class^ a, ConstType b)`.
    It is the LEFT operand, and the op produces self from it. Enforced in
    validateOperatorSignatureTypes (n==2 — the only 2-param shape in the catalog; the swap
    rule is the template). CONSEQUENCE: a binary dispatches on the LHS OPERAND's class and
    NOTHING else — `tryLowerClassBinary` now takes only `lref`. An arbitrary `A:op+(B, C)`
    is rejected (the "worm-can": a class whose op+ takes two unrelated types could only ever
    be selected by the assignment TARGET's type).

    WHY IT MATTERS — OPERATOR PRECEDENCE. `+` binds tighter than `=`, so in `a = b + c` the
    binary must have a meaning fixed by its OWN operands BEFORE the `=` is consulted. Two
    mechanisms violated that and are DELETED (2026-07-11):
      - tryLowerClassBinary's CONTEXT arm — picked the result class from the expected type
        (the `=` target), which is what made a cross-class `Mat m = v1 + v2` expressible.
        Also its rhs-operand arm (a silent reversed-operand path).
      - tryLowerBinaryChain (+ flattenLeftChain, referencesEntry) — the target-keyed chain
        fuse. It ran BEFORE classify's infer loop precisely because it had no operand types
        yet, so it keyed on the TARGET; it re-associated `((m+n)+r)` into
        `a.op+(m,n); a.op+=(r)`, and it CLOBBERED `a` mid-expression (an operand evaluated
        later that reads `a` — e.g. `a = b + c + f()` — sees the half-built accumulator; the
        `referencesEntry` aliasing check could never catch an indirect read).
    Every class binary lowers bottom-up on its own operands. Aliasing (`a = a + b`) works for
    free — the accumulator reads the OLD `a`.

    EXPRESSION-TEMP ELIMINATION ("stage 3 proper") — LANDED 2026-07-12. A chain costs the
    MINIMUM number of temporaries, and a survivor dies at the SEMICOLON. classify STAMPS the
    resolved operator candidates on the kBinaryExpr (stampClassBinary replaced
    tryLowerClassBinary; classify mints no operator temp); DESUGAR lowers the chain and owns
    every temp decision, because the ELIDE needs the chain and its destination together and
    only a whole statement has both. The accumulator's home is the declarator funnel's
    EXISTING "raw storage vs live object" question: a declaration (or a return, via NRVO) of
    the exact class IS the accumulator — zero temps; a LIVE target gets a statement-scoped
    temp MOVED in (through a CALL to the class's op<--, not a bare move node). The ladder:
    a head construction collapses into the accumulator; the first operand seeds with op= on a
    default accumulator and with op<OP>= on one built with args; a real head pair takes the
    2-arg op<OP> or decomposes; later operands FUSE with their own op<OP>=. Result: evaluate.sl
    case J went 12 objects -> 7. Full detail in readme-classes.txt "CLASS-OPERATOR CHAINS";
    the plan (with the decisions and their rationale) in plan-evaluate.txt.
    THE HEAD COLLAPSE IS AN OPTIMIZATION, SO IT CAN BE DECLINED (2026-07-13). `C(7) + a` was
    REJECTED — "Operator '+' is not defined on class 'C'" — for a class with the 2-arg op+ and
    no op+=, though the operator IS defined and the identical rvalue dispatched fine as a CALL
    head (`mk() + a`) or as the RIGHT operand (`a + C(7)`). The collapse builds the head
    construction straight INTO the accumulator, and an accumulator can only be ADVANCED by an
    operator that mutates it with one argument; with no op<OP>= the chain gave up, never
    considering the 2-arg operator sitting right there, resolved. It now declines the collapse
    and the construction becomes an ordinary rvalue operand (one temp, dead at the semicolon —
    what the right-hand spelling always cost). THE REASON IT COULD NOT DECLINE BEFORE: desugar
    RE-DERIVED the role from the AST (lhs.is_construction) instead of reading a stamp, so
    classify had no way to tell it. The role is now decided once, in classify, and carried on
    the node (`op_collapse_head`) — the same discipline every other chain decision already
    followed. Canon operator.sl (class Head — all three spellings agree) + evaluate.sl Z8/Z8b
    (the declined collapse costs ONE temp; K3 still pins the collapse being TAKEN).
    THE ARITY-1 UNARY JOINED IT (2026-07-13). `-a` produces a whole class value, so it is a
    chain node too: stampClassUnary stamps the kUnaryExpr (op_un_eid = the 1-arg op, op<--,
    the default field tuple) and desugar places the accumulator — fresh storage IS it (zero
    temps), a live target gets one temp moved in through op<--, and a unary at a chain HEAD
    collapses into the accumulator like a head construction. Dispatch is on the OPERAND's

    A TUPLE OF CLASSES IS EVALUATED INTO THE STORAGE THAT OWNS IT (2026-07-13) — the last of
    the temp-elimination work, and it landed as a SOURCE MODEL, not as new machinery. Two
    halves. (1) A tuple LITERAL is taken APART: in a position that owns RAW storage (a decl, a
    return, a field of something being constructed), desugar DISTRIBUTES it slot by slot and a
    construction element is built DIRECTLY in its slot — `((C(1), C(2)), C(3))` costs THREE
    constructions, not three temps plus three copies. It generalized past tuples: a
    construction as a class FIELD and inside a RETURNED tuple build in place for the same
    reason. A LIVE target is excluded — its slots are already objects, so each element is
    constructed as a statement-scoped temp and TRANSFERRED in. (2) THE DESTRUCTURE'S SOURCE is
    now an EXPRESSION, cloned and indexed per slot (isReReadableLvalue = an lvalue that is also
    safe to re-read), spilled ONLY when it must be evaluated once (an rvalue, or a literal that
    reads a target — `(sa, sb) = (sb, sa)` must swap). NESTED MOVE/SWAP fell out of that with
    no new code: `src[i]` is real storage, so `<--` nulls the real leaf and `<-->` exchanges
    with it at any depth. The old ident-only source test was stricter than its own reason. Cost:
    block P's 4 extra objects -> 0 (op<-- finally runs there), a nested copy 5 objects -> 3.
    It also closed a TRANSFER-INVARIANT hole: a tuple literal of class LVALUES used to build one
    whole struct value and store it — a BLIT past every element's op= (readme-classes.txt).
    Canon evaluate.sl block V.
    A CLASS CAN ONLY BE COPIED INTO (2026-07-13) — so it must EXIST first: every binding is
    alloc, init, ctor, THEN the transfer. The funnel does NOT produce that order on its own —
    it FILLS storage and then FINALIZES it — so a fresh decl from an lvalue had its CONSTRUCTOR
    RUN ON TOP OF THE COPY and threw it away: `D dx = dy;` on a class whose ctor writes its own

    flattenScope lifts each opening's methods. EXTERNAL: `int C:m()` (method), `C:op+=(…)`
    (operator), `C:_()` / `C:~()` (ctor/dtor — a method with restrictions takes every method
    form; class-only, so a namespace target is rejected in relocation), `C:Ns {}`
    (namespace), `C:R() {}` (hoisted-class re-open, EMPTY parens), `const int C:k=7;`,
    `alias C:A=int;`, and `enum int C:E ( … )` (NAMED enum, members reached qualified). A
    field-bearing `C:X(fields){}` STAYS inheritance (token-identical to `Base:Derived`).
    grammar tags node->qualifier (parseFunctionDef head + looksLikeQualifiedScopeDef/
    parseQualifiedScopeDef + parseEnumDecl qualified name; tail `{`/`()`/`(fields)` dispatching
    namespace/re-open/inheritance BEFORE looksLikeClassDef). resolve.relocateOutOfLineMembers
    runs PER-SCOPE (program->children + registerScopeNames + resolveStmtList/resolveFunctionBody),
    so the external form works in ANY scope the class is DECLARED in — file, namespace body,
    class body, function body/nested block. It walks the qualifier scope-in-scope through classes
    AND namespaces (all openings searched) and MOVES the node under the LOCAL target sibling — a
    method into a CLASS gets `_$recv` HERE (the parser skips the receiver for a qualified method
    in a class body, else it doubles); a namespace/class node or namespace free function gets
    none. EMPTY-FIELD DERIVED: the empty-parens head `A:B() {}` is token-identical to an
    external re-open, so grammar always routes it as a re-open, then relocation decides
    SEMANTICALLY — if A is a CLASS whose openings do not already contain B, the node is
    reinterpreted in place as INHERITANCE (`A : B()`: set ->text, prepend the `_$base` slot-0
    field, clear the qualifier); B-in-A -> re-open, A-a-namespace -> create nested. So an
    empty-field derived class is now spellable; creating a NEW nested class inside a class via
    the external empty-parens form is not (write a block). SAME-SCOPE for classes: the target
    must be a local sibling, so re-opening a class merely VISIBLE from an enclosing scope
    (refine) fails the walk and errors per-segment. NAMESPACES
    open in any scope: a qualified LEAF whose first segment names an enclosing-scope namespace is
    registered into that frame IN PLACE (registerQualifiedLeaf, node left for constfold; the
    intermediate resolve passes skip a qualified leaf). A qualified MUTABLE var is rejected
    ("Only constants, aliases, and enums may be defined by qualified name"). Chained
    (`C1:Ns1:C2:Ns2:C3:m`) + external-ns merge both work. The cross-scope run-time variant is
    split into REFINEMENTS below. Canon test/class/reopen.sl (Stages A-F: block/method/
    namespace/hoisted-reopen/chained + const/alias/enum in file/namespace/class/function scope;
    field-add / redefine / leak / bad-path / refine-reject negatives). [[project_reopen_classes]]
  * A CTOR/DTOR IS A METHOD WITH RESTRICTIONS — LANDED 2026-07-16. The governing rule (canon):
    a ctor/dtor is a method with restrictions, LIKE AN OPERATOR, so every syntax a method admits
    it admits too. The restrictions are only: no params, no return type, class-only, ctor never
    virtual. What this closed: the EXTERNAL out-of-line form `C:_()` / `C:~()`, chained
    (`A:B:~()`) and in every scope the class is declared (Stage E falls out — relocation is
    already per-scope). Five scans walk a `:`-chain and each had to stop before the hook, so they
    share ONE test (isHookHead): parseQualifiedName (as it already did for `:op`),
    looksLikeQualifiedScopeDef and looksLikeClassDef — BOTH of which silently MATCHED `C:_() {}`,
    as a hoisted-class re-open of a class named `_` and as an empty-field derived class named `_`
    (token-identical shapes; the hook wins, as the class-body loop already had it), plus
    looksLikeFunctionDef and parseFunctionDef's qualifier loop. Restrictions enforced for the
    out-of-line spelling exactly as inline: no return type, no params, and NAMESPACE-TARGET

    segments (token-count checked, 2n-1, so segment->token stays exact). Hooks and operators were
    one fix. Canon reopen.sl: `Ext` / `Hh:Nst` / `Hs:Veg` / `Hc2:Sib` positives (external,

  * CLASS LIFECYCLE ACROSS OPENINGS — LANDED 2026-07-16. The ctor/dtor contract is per CLASS,
    not per BODY — the lifecycle is the union of every opening, like every other member. A
    `_();` / `~();` forward declaration may sit in the primary with its DEFINITION in a re-open,
    the two halves may land in two SEPARATE re-opens, or a re-open may add both hooks with no
    declaration anywhere. A declaration now emits a bodyless kFunctionDecl member
    (`_$ctor`/`_$dtor`) instead of nothing — every downstream hook scan already accepted one
    (resolve's type pass even says hooks "have no entry but still need their `_$recv` typed"),
    and flattenScope lifts only kFunctionDef, so no stray symbol appears. Both the PAIRING rule
    and the must-be-defined obligation moved OUT of parseClassDef (it reads one body at a time
    and structurally cannot see a later opening) into registerClassBody, which scans the
    primary's own members PLUS ClassInfo.pending_hooks — the re-opens' hooks, carried to the
    primary exactly as pending_fields carries appended fields, and left OWNED by the re-open node
    so the body pass still resolves them there. Pairing GATES the obligation (a lone `_();`
    reports the missing dtor once, not that plus "must be defined"); declaring the pair while
    defining one half now names the missing DEFINITION instead of a phantom pairing violation.
    DUPLICATES joined them (2026-07-16): two definitions of one hook in two OPENINGS were caught
    by nothing — the parser's per-body "Duplicate constructor." saw one body — so slidsc emitted
    two `@C__$ctor__impl` definitions and llc rejected the IR. Now diagnosed per class with the
    "first defined here" note a duplicate METHOD already got; that asymmetry is what exposed it.
    grammar now keeps NO hook state at all: all three rules span openings.
    THE BUG THIS CLOSED WAS SILENT: a re-open's hook was invisible to the primary's lifecycle
    scan, so has_ctor stayed false, the hook was NEVER CALLED, and its `__impl` was emitted DEAD
    — it compiled, linked, ran, and printed nothing. parseGlobal keeps its own per-body pairing
    check (its own `_$gctor`/`_$gdtor` + its own messages): a global group is ONE body and cannot
    be re-opened, so per-body is correct there. Canon test/class/reopen.sl Stage G — positives
    `Forward` (decl in primary, def in a re-open), `Split` (halves in two re-opens), `Late` (a
    re-open adds both, undeclared); each FAILS without the fix, `Late` by printing nothing. Five
    negatives pin the obligation + pairing across openings (regression locks: each produced the
    same message before, from the parser, for the wrong reason).

  * INCOMPLETE CLASSES — LANDED (single-file). A class whose field tuple ends with a trailing
    `...` is INCOMPLETE (Node.is_incomplete; grammar parseParamList, class field lists only,

  * REFINEMENTS (run-time-scope re-open) — PLANNED (stub test/class/refine.sl). Re-opening
    a class from a scope that ISN'T its own is an ILLUSION: it desugars to an anonymous
    zero-field DERIVED class ($T : T) that USURPS the base's name within that lexical scope,
    and existing base instances are cast into it (`(<T:$T^> ^x)^.m()`). Both the block form
    (`Class(){ ... }`) and the external form (`void Class:method(){ }`) trigger it in a run-time
    scope. SOUND because a re-open adds NO fields, so $T is layout-identical to T → the
    base->derived cast is a free offset-0 no-op, and the whole thing rides the LANDED inheritance
    (_$base slot 0, base-chain member lookup). Delivers the scoping asymmetry for free (the
    original + sibling scopes can't name $T, so they never see the added members) and COMPOSES
    WITH TEMPLATES: because the refinement is a real TYPE, `sort(^arr)` in the refining scope
    instantiates `sort<$T>`, whose body finds the refined method — retroactive, scoped
    conformance (slids's answer to a comparator / trait-impl, no orphan-rule problem). MOTIVATION:
    `sort<T>(T^ c){ if (c[j].lessthan(c[i])) c[i] <--> c[j]; }` — refine the element type to
    supply `lessthan` locally. Internal name for $T: the "usurper". [[project_refinements]]
